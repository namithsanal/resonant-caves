## Current state

The project is scaffolded as a Fabric mod targeting **Minecraft 1.21.1** (pinned versions under "Architecture & tech decisions"). **Feature 1 (unbreakable stone & deepslate) is implemented and verified** — bedrock-like (no mining animation) via a mixin plus a server-side veto. **Feature 2 (large fleeing herds) was implemented, verified, and later fully reverted at the player's request** — it originally raised biome spawn group sizes via Fabric's `BiomeModifications` API and injected a `FleeEntityGoal` into sheep/cow/pig/chicken via mixin (horses got larger herds but stayed tame); `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted and the `FleeingAnimalsMixin` entry removed from `resonantcaves.mixins.json`, with no replacement code, so all passive animals are back to vanilla spawn-group sizes and behavior. **Feature 3 (Resonant Ore) is implemented and verified** — two bedrock-hard ore blocks with a block entity that rolls a 0–15 RF/t generation rate and exposes it via the TeamReborn Energy API, generated underground at roughly iron ore's rarity via `BiomeModifications.addFeature`. **Feature 4 (elevated world) went through several iterations and is now implemented as a separate, selectable world preset rather than an override of vanilla's Overworld.** Earlier versions overrode vanilla's actual `noise_settings/overworld.json` + `density_function/overworld/depth.json` directly — shifting sea level to 280, which clipped hills against the still-fixed Y320 build cap since the height envelope itself was never extended. The current design instead adds two coexisting world presets, chosen at world-creation time: **"Vanilla World"** (`minecraft:normal`, relabeled via a lang-key override) is genuinely untouched vanilla generation — `data/minecraft/worldgen/noise_settings/overworld.json` and `data/minecraft/worldgen/density_function/overworld/depth.json` are no longer overridden at all. **"Resonant World"** (`resonantcaves:resonant_world`, registered in `data/resonantcaves/worldgen/world_preset/`) is a new namespaced dimension type + noise settings (`data/resonantcaves/dimension_type/resonant_overworld.json`, `data/resonantcaves/worldgen/noise_settings/resonant_overworld.json`) re-anchored to a symmetric `min_y: -256` / `sea_level: 0` / build cap `256` coordinate system — chosen so the headroom above sea level matches vanilla's proportions almost exactly (256 vs vanilla's 257), eliminating the earlier clipping by construction rather than patching around it, while giving roughly double vanilla's below-sea-level cave volume. Both presets use the registry key `minecraft:overworld` for their Overworld dimension (that's how world presets work — picked once per save, not concurrent dimensions), so every other mod feature (unbreakable stone, Resonant Ore, creepers, the helmet) works identically in both without any changes. Several vanilla values anchored to specific Y coordinates were recomputed for Resonant World by preserving their *distance* from the relevant vanilla reference point, then re-anchoring to the new bounds: the ceiling terrain-dampening gradient (240/256 → 177/193, sea-level-relative), the floor solid-layer gradient (-64/-40 → -256/-232, bottom-relative), the `depth` density function's gradient (now a separate `resonantcaves:overworld/depth` registry entry, -64/320 → -256/256, tracking the dimension's real bounds rather than a "virtual" mismatched range — the root cause of the earlier clipping), the ore-vein Y-band (-60/51 → -252/-141, bottom-relative), the deepslate transition (absolute 0/8 → -192/-184, bottom-relative), and several sea-level-anchored surface-rule decoration thresholds in swamp/mangrove swamp/windswept gravelly hills/wooded badlands biomes (60/62/63/74/97, each shifted by the same -63 as sea level; the unrelated legacy `absolute: 256` badlands terracotta-banding threshold was left untouched, confirmed independent of build height). The bedrock floor surface rule needed no change (`above_bottom`-anchored, already relative). Vanilla's cave-cheese-threshold tweak (`0.27` → `0.15`, denser caves) moved into Resonant World's noise settings only, since Vanilla World is meant to be genuinely vanilla. Cloud height (vanilla 192, 128 below the build cap) is handled in Java since it isn't data-driven per-dimension by default: a client-only `CloudHeightMixin` overrides `DimensionEffects.getCloudsHeight()` to return 128 — guarded by checking the *loaded world's* `DimensionType.minY()` rather than `DimensionEffects` instance identity, since both presets share the `"minecraft:overworld"` effects id. A common-side `WorldPresetsMixin` makes Resonant World the default pre-selected preset by mutating `WorldPresets.DEFAULT` (confirmed safe: it's assigned via a method call, not a compile-time constant, so it's read fresh at each call site rather than inlined). The new preset is made visible in the world-creation menu via an additive merge into vanilla's `data/minecraft/tags/worldgen/world_preset/normal.json` tag (tags merge across mods by default, no full-file override needed). `./gradlew build` is green after this change. **After an in-world playtest of Resonant World, its caves turned out to be flooded with lava almost everywhere.** The root cause was two hardcoded absolute-Y constants in vanilla Java code (not exposed via any noise_settings JSON, so the earlier data-only re-anchoring missed them): `NoiseChunkGenerator.createFluidLevelSampler`'s default fluid-type decision (`y < Math.min(-54, seaLevel) ? LAVA : WATER` — in vanilla only the bottom 10 blocks above its min_y of -64, but with Resonant World's floor at -256 this covered 202 of 256 blocks below sea level, the dominant effect) and `AquiferSampler.Impl.getFluidBlockState`'s lava-pocket-noise threshold (`fluidLevel <= -10`, vanilla's same 54-blocks-above-the-floor pattern, a smaller secondary effect). Both were fixed with new mixins — `NoiseChunkGeneratorLavaLevelMixin` (the real fix, `@ModifyConstant` re-deriving `-54` as `settings.generationShapeConfig().minimumY() + 10`) and `AquiferLavaThresholdMixin` (captures the dimension's `minimumY` via a constructor `@Inject`, then `@ModifyConstant`s `-10` to `minimumY + 54`) — both formulas reproduce vanilla's exact behavior at vanilla's bounds (-64 + 10 = -54, -64 + 54 = -10) while scaling correctly for any other dimension height. An initial version of the `NoiseChunkGeneratorLavaLevelMixin` handler had its parameter order backwards (Mixin requires the modified constant's type first, then captured target-method args — not the reverse), which crashed the dedicated server outright at boot; caught via a headless smoke test before being shipped. Verified compiling + a headless smoke test that exercises real chunk/aquifer generation (not just mod/mixin loading) for Resonant World; this only affects newly-generated chunks, so previously-explored saves keep their already-placed lava. **Feature 5 (creeper tech-mod targeting) is implemented and verified** — a new `resonantcaves:creeper_attracting_blocks` block tag (`data/resonantcaves/tags/block/`, ~40 Tech Reborn energy-infrastructure blocks — storage units, transformers, cables, solar panels, generators — all `required: false`) lets creepers sense and beeline toward tagged blocks via `SeekAndDetonateGoal`, injected by `CreeperTargetingMixin`; on arrival the creeper calls the vanilla `ignite()`, so detonation is an ordinary power-3 explosion with no custom destruction logic. Verified compiling + a headless server smoke test (AI/tag-only change, no world deletion needed); in-world `runClient` testing with Tech Reborn installed is the natural follow-up. **Feature 6 (creeper pack spawning) is implemented and verified** — after two attempted fixes to a group-size-based approach (raising `SpawnSettings.SpawnEntry`'s creeper group size to 10–15 via `HerdSpawning`'s `growHerd`, then also overriding `CreeperEntity.getLimitPerChunk()` to 15) still produced no visibly larger packs in-world, a registry-level diagnostic (logged at server start, no playtest needed) confirmed both were correctly applied — every overworld biome had `creeper*(10-15):100` and `getLimitPerChunk()` returned 15 — so the real issue is vanilla's group-spawn algorithm itself: each pack member is placed via its own independent ±5-block random walk from the initial roll position, which mostly fails `canSpawn`'s space checks in this mod's narrow, unbreakable-walled cave tunnels. Both the group-size rewrite and the `getLimitPerChunk` override were reverted, and replaced with `CreeperPackSpawning`: on `ServerEntityEvents.ENTITY_LOAD`, any newly-spawned creeper becomes a pack "leader" and spawns 9–14 more creepers within 3 blocks of it, each candidate position checked with `World.isSpaceEmpty` before placement, with all pack members (including the leader) tagged `resonantcaves_creeper_pack` so chunk reloads of existing creepers don't repeatedly spawn new packs around them. Verified compiling + a headless server smoke test; in-world `runClient` confirmation of pack size/feel is the natural follow-up. `./gradlew build` is green and a headless server smoke test loads the mod + mixins + worldgen cleanly. Compiling requires a JDK 21 (see "Building & running"). **The mod was renamed from "Surface Survival" (id `surfacesurvival`) to "Resonant Caves" (id `resonantcaves`, package `com.namith.resonantcaves`)** — a full rename across the Java package, mod ID, resource namespaces (`data/resonantcaves/`, `assets/resonantcaves/`), mixins config, and Gradle project/group, with no behavioral changes. The Resonant Ore textures were also recolored to a coral/salmon "orangeish-pink" palette (derived from vanilla gold ore), and the mod's icon is now an upscaled `deepslate_resonant_ore` texture. **Feature 7 (Resonant Helmet) is implemented and verified** — `ModItems` registers a custom `resonantcaves:resonant` `ArmorMaterial` (gold's defense/enchantability/equip-sound/toughness/knockback-resistance, but with iron's durability multiplier, giving the helmet 165 durability) and the `resonantcaves:resonant_helmet` `ArmorItem`, craftable via a shaped recipe (`GSG`/`EHE`/`III` = gunpowder/spider eye/ender pearl/iron helmet/gold ingot, `data/resonantcaves/recipe/resonant_helmet.json`). `NightVisionRefresher` (server tick, every 100 ticks) re-applies amplifier-0 Night Vision (320-tick duration, no particles/icon) to any player wearing the helmet so it never flickers off. A new `ResonantCavesClient` client entrypoint (added to `fabric.mod.json`) registers `ClientTickEvents.END_CLIENT_TICK` to spawn a slowly "breathing" (sine-pulsed) `ParticleTypes.GLOW` orb at the center of every `HostileEntity` within 128 blocks of a helmet-wearing player — a hostile-mob radar visible only to that player. The helmet's textures (16x16 item icon + 64x32 armor layer) were derived from the recipe's own ingredient palette (gunpowder grays, ender-pearl teal "energy lens", a glowing red "eye" accent at the former specular-highlight pixels) via a Pillow remap of the vanilla gold-armor textures. Verified compiling + a headless server smoke test (recipe count increased 3766→3767). After an in-world playtest, several follow-up refinements were made to Feature 7's Night Vision and radar effects. `NightVisionRefresher` was rewritten to track each player's helmet-wearing state in a `WEARING` set: Night Vision is now applied/removed the instant a player equips/unequips the helmet (previously there was a delay until the next 100-tick refresh), with `EFFECT_DURATION_TICKS` raised 220→320 — vanilla's `GameRenderer.getNightVisionStrength` pulses the screen-darkness fade once the remaining duration drops below 200 ticks, causing a visible "flicker" near each refresh, and 320 keeps the minimum remaining duration safely above that threshold. A new client-only `NightVisionStrengthMixin` caps the value returned by `GameRenderer.getNightVisionStrength()` to 0.1 for helmet wearers, making the helmet's Night Vision noticeably weaker/dimmer than the vanilla potion (tuned interactively: 0.5→0.15→0.05→0.1 final). A new client-only `StatusEffectIconMixin` (registered under the `client` list in `resonantcaves.mixins.json`, since `AbstractInventoryScreen` doesn't exist on a dedicated server) filters `AbstractInventoryScreen.drawStatusEffects`'s effect list down to `shouldShowIcon()`-true effects, so the inventory-screen effect panel no longer shows Night Vision (the top-right HUD already respected `showIcon=false` via `InGameHud`). Separately, `ResonantCavesClient` was rewritten so the radar is visible through walls: instead of spawning `ParticleTypes.GLOW` particles on `ClientTickEvents.END_CLIENT_TICK` (depth-tested and occluded by walls), it now hooks `WorldRenderEvents.AFTER_TRANSLUCENT` and draws a manual billboard quad per hostile mob via `RenderLayer.getTextSeeThrough(...)`, textured with a new 32x32 radial-gradient PNG (`assets/resonantcaves/textures/misc/radar_orb.png`). Since `getTextSeeThrough`'s "ALWAYS_DEPTH_TEST" phase is actually a no-op for `GL_ALWAYS` (Mojang special-cases `depthFunction == 519` to skip touching GL state), the draw is wrapped in explicit `RenderSystem.disableDepthTest()`/`enableDepthTest()` calls so the orb truly renders through walls. Each orb's quad size scales with distance to the camera (`ANGULAR_SIZE = 0.004`) so it stays a small, constant size on screen regardless of range; a shared 300-tick (~15s) breathing cycle drives both opacity (fading between 15% and ~78%) and color (fading between dark teal, matching an ender pearl, and a light bluish-green matching vanilla's `GLOW` particle); and each mob gets a UUID-seeded per-axis frequency/phase offset so its breathing and a small (±0.25 block) wander around its center are out of sync with other mobs' but synced with each other. Verified compiling + an in-world `runClient` session iterating on all of the above with no new errors. **The Resonant Helmet's crafting recipe was later removed entirely** and replaced with a drop mechanic: `ZombieHelmetSpawning` gives plain zombies a 5% chance to spawn wearing the helmet, with its head-slot drop chance set to `2.0F` (vanilla's "always drop" convention, per `PatrolEntity`'s banner) so it's guaranteed to drop on death regardless of cause. Verified compiling + a headless server smoke test (recipe count back to baseline 3766). **The helmet's textures were then redone** as a recolor of the vanilla turtle shell helmet (rather than the original gold-armor remap) — see Feature 7 below for the exact hue mapping. Separately, **K-Turrets was evaluated as a Feature 5 tag candidate and rejected**: its turrets are entities (`entity.k_turrets.*_turret`), not blocks, so they can't be referenced by a block tag; vanilla's `RevengeGoal` already makes creepers retaliate against whatever last damaged them (including a turret), so no new code was added for it. The only real K-Turrets *block* (`k_turrets:reloader`) was deliberately left out of the tag per a direct ask.

## What this project is

**Resonant Caves** is a single Fabric mod for Minecraft Java Edition implementing a "Surface Survival" overhaul. The core fantasy: **mining is impossible**, so the game becomes about exploration, hunting, and infrastructure instead of digging.

- New worlds default to the **"Resonant World"** preset — a re-anchored coordinate system (sea level 0, build cap 256, world bottom -256) giving roughly double vanilla's below-sea-level cave volume as *traversal* space, not mineable rock, while keeping vanilla's proportions above sea level. A genuinely untouched **"Vanilla World"** preset is also selectable for comparison.
- Stone and deepslate are **unbreakable** (like bedrock); everything else — including stone variants (granite, diorite, andesite, tuff), dirt, gravel, and all ores — stays breakable, so resources remain accessible only where exposed in cave walls.
- A naturally-occurring unbreakable ore ("Resonant Ore") passively emits a small amount of RF, so energy comes from finding and claiming cave nodes rather than tech-mod machines.

An earlier feature — large fleeing herds — was built, played, and then fully reverted at the player's request; see Feature 2 below for what's left of them in the repo.

## Hard scope discipline (read before adding anything)

This is a ~20-hour, AI-assisted, *first* mod. The design explicitly rejects features that become rabbit holes. **Do NOT build:**

- A custom chunk generator from scratch — modify vanilla `noise_settings` instead.
- Custom pathfinding or terrain-safety navigation — vanilla pathfinding is acceptable for v1.
- A rewritten mob AI system or true flocking/boids — fake herd behavior with large spawn groups + a strong flee radius.
- Migration/sleeping/predator-prey systems.
- Tech-mod infrastructure: machines, power networks, or internal energy storage/buffering.

**Do:** keep implementation simple, lean on vanilla systems with small tweaks, and use existing APIs.

The `resonantcaves:creeper_attracting_blocks` tag (Feature 5) is the one sanctioned point of integration with other mods' blocks — it's a data file, not new infrastructure, so adding entries for another tech mod doesn't violate the "no tech-mod infrastructure" rule above.

`item_obliterator.json5` (repo root) is **not part of the mod build** — it's a config file for the separate, player-installed [Item Obliterator](https://github.com/nvb-uy/Item_Obliterator) mod, distributed alongside this mod's jar so players can blacklist Tech Reborn's other power sources (solar panels, generators, etc.) and force reliance on Resonant Ore instead. See README.md's "Requirements" for the full recommended mod list (Tech Reborn, K-Turrets).

## The features

Features 1-4 were the original MVP; build those in order, with **worldgen last** since debugging it can consume the entire budget. Features 5 and 6 are post-MVP additions built after an in-world playtest. **Feature 2 was later fully reverted at the player's request** (see their entries below and "Current state" for what was undone and what survived).

1. **Unbreakable stone & deepslate** *(Easy — built ✓)* — `minecraft:stone` and `minecraft:deepslate` only; all other blocks (stone variants granite/diorite/andesite/tuff, dirt, gravel, ores) stay breakable. Implemented in two layers sharing `UnbreakableBlocks.isUnbreakable`: (a) `AbstractBlockStateMixin` forces `calcBlockBreakingDelta` to `0` so survival players get the **bedrock feel** — no cracking animation, no progress; (b) a `PlayerBlockBreakEvents.BEFORE` handler is an authoritative server-side veto. Creative players are exempt (they break via a separate instant path). Verified compiling + a headless server smoke test.
2. **Large fleeing herds** *(Easy — built, then reverted ✗)* — originally, `HerdSpawning` used Fabric's `BiomeModifications` API to rewrite spawn group sizes for sheep/cows/pigs/chickens/horses, and `FleeingAnimalsMixin` injected a `FleeEntityGoal` + `StayNearHerdGoal` into sheep/cow/pig/chicken via mixin. Fully reverted at the player's request: `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted, the `FleeingAnimalsMixin` mixins.json entry removed, and the `HerdSpawning.register()` call removed from `ResonantCaves.onInitialize()`. No replacement code — animal spawning and AI are pure vanilla again.
3. **Resonant Ore** *(Medium — built ✓)* — `ModBlocks` registers `resonant_ore` (stone variant) and `deepslate_resonant_ore`, both bedrock-hard via `AbstractBlock.Settings.strength(-1.0F, 3600000.0F)` — naturally unmineable in any game mode, no mixin needed (`UnbreakableBlocks.isUnbreakable` lists them too, for a single source of truth). Only `resonant_ore` gets a `BlockItem`/creative-tab entry, since neither block is ever obtainable in survival. `ResonantOreBlockEntity` rolls a single `generationRate` (0–15 RF/t) from a triangular distribution peaking at 4, seeded deterministically from the block's position (and persisted to NBT). `ResonantEnergyStorage` is a stateless, extract-only `team.reborn.energy.api.EnergyStorage` (refuses insertion; `getAmount`/`getCapacity`/`extract` all read `generationRate` directly) registered via `EnergyStorage.SIDED.registerForBlockEntity`. Worldgen is a `minecraft:ore` configured feature + placed feature (`data/resonantcaves/worldgen/...`, count 52, size 6, full-world-height `above_bottom: 0` to `below_top: 0` placement so generation rate is consistent at every Y level regardless of which world preset is active) registered with `BiomeModifications.addFeature` at `GenerationStep.Feature.UNDERGROUND_ORES` — roughly iron ore's rarity, may be buried (`discard_chance_on_air_exposure: 0.0`). Verified compiling + a headless server smoke test.
4. **Two world presets: "Resonant World" (default) and "Vanilla World"** *(Medium — built ✓)* — see "Current state" above for the full history and exact Y-value table. `data/minecraft/worldgen/noise_settings/overworld.json` / `density_function/overworld/depth.json` are no longer overridden, so **"Vanilla World"** (`minecraft:normal`) is genuinely stock vanilla generation. **"Resonant World"** (`resonantcaves:resonant_world`) is a new namespaced dimension type + noise settings re-anchored to `min_y: -256` / `sea_level: 0` / build cap `256`, registered as a `world_preset` and made the pre-selected default via `WorldPresetsMixin`. The two presets are display-renamed via lang-key overrides (`generator.minecraft.normal` → "Vanilla World", `generator.resonantcaves.resonant_world` → "Resonant World") and both appear in the main (non-shift) preset list via an additive tag merge. `./gradlew runClient` + in-world exploration of both presets (no clipping, sensible cloud height, normal-feeling caves/ores/deepslate/fluids in Resonant World; identical-to-vanilla in Vanilla World) is the natural follow-up.
5. **Creeper tech-mod targeting** *(Easy — built ✓)* — Resonant Caves defines a block tag, `resonantcaves:creeper_attracting_blocks` (`data/resonantcaves/tags/block/creeper_attracting_blocks.json`), pre-populated with ~40 Tech Reborn energy-infrastructure blocks (storage units, transformers, cables, solar panels, generators) as `required: false` entries — other tech mods can be supported by adding their block IDs to this same tag (via this mod or a small datapack), with no Java changes needed. `CreeperTargetingMixin` injects a `SeekAndDetonateGoal` into every creeper's `initGoals` (priority 4, alongside `MeleeAttackGoal`): the goal scans a 24×6×24-block box around the creeper for the nearest tagged block (`BlockPos.findClosest`), paths to it with vanilla navigation, and once within ~1.5 blocks calls the creeper's public `ignite()` — the existing fuse/explosion code then runs unmodified, so detonations are ordinary power-3 creeper explosions. A 30-second give-up timer abandons unreachable targets (relevant since stone/deepslate are unbreakable, so a walled-off machine room may be unreachable). This makes Tech Reborn's energy infrastructure — which the player will use to draw RF from Resonant Ore — a standing threat that rewards defensive placement rather than a "guaranteed destroy" mechanic. Verified compiling + a headless server smoke test; in-world `runClient` testing with Tech Reborn installed is the natural follow-up.
6. **Creeper pack spawning** *(Easy — built ✓)* — `CreeperPackSpawning` listens for `ServerEntityEvents.ENTITY_LOAD`: any newly-spawned creeper becomes a pack "leader" and immediately spawns 9–14 more creepers within 3 blocks of it (each candidate position checked with `World.isSpaceEmpty` before placement), with all pack members — leader included — tagged `resonantcaves_creeper_pack` so a later chunk reload of an existing creeper doesn't spawn another pack around it. Combined with Feature 5, a pack converging on a player's Tech Reborn setup means multiple independent `SeekAndDetonateGoal` instances can detonate near the same infrastructure at once, amplifying the "defend your base" threat. **History:** the original approach (`HerdSpawning`'s `growHerd` raising the creeper `SpawnSettings.SpawnEntry` group size to 10–15, plus a `CreeperTargetingMixin.getLimitPerChunk()` override to 15) was verified correct at the registry level but produced no visible packs in-world — vanilla's per-member ±5-block random-walk placement mostly fails in this mod's narrow cave tunnels — so it was reverted in favor of the clustered-spawn approach above. Verified compiling + a headless server smoke test; in-world `runClient` confirmation is the natural follow-up.
7. **Resonant Helmet** *(Medium — built ✓)* — `ModItems` registers a custom `resonantcaves:resonant` `ArmorMaterial` (gold-tier defense/enchantability/equip-sound/toughness/knockback-resistance, iron's durability multiplier → 165 durability) and the `resonantcaves:resonant_helmet` `ArmorItem`. While worn, `NightVisionRefresher` (server tick) keeps amplifier-0 Night Vision permanently topped up (capped to 0.1 screen-effect strength by the client-only `NightVisionStrengthMixin`, and hidden from the inventory effects panel by `StatusEffectIconMixin`), and `ResonantCavesClient` renders a "hostile mob radar": a manually-drawn, depth-test-disabled billboard quad (`textures/misc/radar_orb.png`) hovers at the center of every hostile mob within 128 blocks, visible through walls, at a constant on-screen size regardless of distance. Each orb slowly "breathes" — a 300-tick cycle drives both its opacity (15%–~78%) and a color fade between dark teal (ender pearl) and light bluish-green (vanilla `GLOW` particle), and a small wander offset synced to that same cycle. Both the breathing phase and wander shape are seeded per-mob from its UUID, so orbs from the same pack desync rather than pulsing in unison. Textures were originally hand-derived from the recipe's own ingredient colors via a Pillow remap of vanilla gold-armor textures, then later replaced by a remap of the vanilla **turtle shell helmet** textures (16x16 item icon + 64x32 armor layer): the bright-green scute-highlight pixels (hue ~121°) become a coral/salmon ramp matching Resonant Ore's palette, the duller olive-green shell-base/AO-shading pixels (hue ~133–143°) become a gold ramp, and the brown strap pixels are left untouched. **The helmet is no longer craftable** — `ZombieHelmetSpawning` listens for `ServerEntityEvents.ENTITY_LOAD` and gives plain `ZombieEntity` spawns (husks/drowned/zombie villagers/zombified piglins excluded via an exact `getClass() == ZombieEntity.class` check) a 5% chance to spawn already wearing the helmet, tagging every rolled zombie `resonantcaves_zombie_helmet_roll` so chunk reloads don't re-roll. The helmet's head-slot equipment drop chance is set to `2.0F`, mirroring vanilla's pillager-captain banner (`PatrolEntity`) — `MobEntity.dropEquipment` treats any drop chance `> 1.0F` as an unconditional drop, so the helmet always drops on death regardless of what killed the zombie (player, mob, fall, fire, etc.), not just player kills. Verified compiling + a headless server smoke test (recipe count back to baseline 3766, confirming the recipe removal and no other regressions); in-world `runClient` confirmation of spawn rate/feel is the natural follow-up.

## Architecture & tech decisions

- **Loader:** Fabric (Fabric Loom + Fabric API). Start from the [Fabric Example Mod](https://github.com/FabricMC/fabric-example-mod) template.
- **Language/tooling:** Java, IntelliJ IDEA.
- **Minecraft version (pinned): 1.21.1**, with **Yarn mappings** (`yarn 1.21.1+build.3`), **Java 21**, Fabric Loader `0.19.3`, Fabric API `0.116.12+1.21.1`, Loom `1.16-SNAPSHOT`, Gradle `9.4.1`. We deliberately chose the established 1.21.1 over the newest Minecraft (26.x): version mismatch — not coding — is the biggest risk, and 1.21.1 has the deepest tutorial/example base, the most mature library support (including TeamReborn Energy), and well-settled tooling. We use **Yarn** rather than Mojang mappings (the current example-mod template defaults to Mojang) because the 1.21.1 tutorial/example ecosystem is overwhelmingly Yarn-based. **Match all docs/examples to 1.21.1 + Yarn.**
- **Loom plugin gotcha:** Loom 1.16 splits the Gradle plugin. The plain `net.fabricmc.fabric-loom` plugin does **not** expose the `mappings`/`modImplementation` DSL (it has a mappings-less workflow). For the classic Yarn setup (`mappings "net.fabricmc:yarn:…:v2"` + `modImplementation`), use **`net.fabricmc.fabric-loom-remap`** — which is what `build.gradle` declares.
- **Worldgen lives as datapack-style JSON bundled inside the mod jar** (noise settings, world preset, configured/placed ore features) — not a separate datapack and not a Java chunk generator. This keeps the deliverable a single mod file ("install mod and play") while minimizing Java code.
- **AI goal injection** uses SpongePowered Mixin.
- **Energy** uses the **TeamReborn Energy API** (the mature Fabric-native equivalent of Forge RF). The ore only *reports available extraction and refuses insertion* — energy availability is computed on demand, so no save data or sync is needed.

## Building & running

The project is scaffolded (Loom + Fabric API) and builds successfully. Standard workflow:

```bash
./gradlew genSources    # generate decompiled, Yarn-mapped Minecraft sources for reference
./gradlew runClient     # launch a dev client to test in-game
./gradlew runServer     # launch a dev server
./gradlew build         # compile + produce build/libs/resonantcaves-<ver>.jar
```

**Gradle must run on a JDK 21** (Loom compiles and runs Minecraft 1.21.1 against Java 21; there is no Gradle toolchain configured, so Gradle uses the JVM that launches it). Fedora 44 ships no JDK 21 package (only 25/26), so this repo uses a user-local **Temurin JDK 21** at `/home/namith/jdks/jdk-21.0.11+10`. Invoke Gradle with that JDK:

```bash
JAVA_HOME=/home/namith/jdks/jdk-21.0.11+10 ./gradlew build
```

(Or set IntelliJ's Gradle JVM to that JDK.) There is no test-suite convention yet; add one when tests are introduced.

## Development workflow

Build **one feature at a time**, fully closing the loop each time: verify it compiles → launch the game → test the behavior in-world → commit → move to the next feature. Get a working mod (unbreakable stone) *before* touching worldgen.
