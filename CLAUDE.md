## Current state

The project is scaffolded as a Fabric mod targeting **Minecraft 1.21.1** (pinned versions under "Architecture & tech decisions"). **Feature 1 (unbreakable stone & deepslate) is implemented and verified** — bedrock-like (no mining animation) via a mixin plus a server-side veto. **Feature 2 (large fleeing herds) was implemented, verified, and later fully reverted at the player's request** — it originally raised biome spawn group sizes via Fabric's `BiomeModifications` API and injected a `FleeEntityGoal` into sheep/cow/pig/chicken via mixin (horses got larger herds but stayed tame); `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted and the `FleeingAnimalsMixin` entry removed from `resonantcaves.mixins.json`, with no replacement code, so all passive animals are back to vanilla spawn-group sizes and behavior. **Feature 3 (Resonant Ore) is implemented and verified** — two bedrock-hard ore blocks with a block entity that rolls a 0–15 RF/t generation rate and exposes it via the TeamReborn Energy API, generated underground at roughly iron ore's rarity via `BiomeModifications.addFeature`. **Feature 4 (elevated world) went through several iterations and is now implemented as a separate, selectable world preset rather than an override of vanilla's Overworld.** Earlier versions overrode vanilla's actual `noise_settings/overworld.json` + `density_function/overworld/depth.json` directly — shifting sea level to 280, which clipped hills against the still-fixed Y320 build cap since the height envelope itself was never extended. The current design instead adds two coexisting world presets, chosen at world-creation time: **"Vanilla World"** (`minecraft:normal`, relabeled via a lang-key override) is genuinely untouched vanilla generation — `data/minecraft/worldgen/noise_settings/overworld.json` and `data/minecraft/worldgen/density_function/overworld/depth.json` are no longer overridden at all. **"Resonant World"** (`resonantcaves:resonant_world`, registered in `data/resonantcaves/worldgen/world_preset/`) is a new namespaced dimension type + noise settings (`data/resonantcaves/dimension_type/resonant_overworld.json`, `data/resonantcaves/worldgen/noise_settings/resonant_overworld.json`) re-anchored to a symmetric `min_y: -256` / `sea_level: 0` / build cap `256` coordinate system — chosen so the headroom above sea level matches vanilla's proportions almost exactly (256 vs vanilla's 257), eliminating the earlier clipping by construction rather than patching around it, while giving roughly double vanilla's below-sea-level cave volume. Both presets use the registry key `minecraft:overworld` for their Overworld dimension (that's how world presets work — picked once per save, not concurrent dimensions), so every other mod feature (unbreakable stone, Resonant Ore, creepers, the helmet) works identically in both without any changes. Several vanilla values anchored to specific Y coordinates were recomputed for Resonant World by preserving their *distance* from the relevant vanilla reference point, then re-anchoring to the new bounds: the ceiling terrain-dampening gradient (240/256 → 177/193, sea-level-relative), the floor solid-layer gradient (-64/-40 → -256/-232, bottom-relative), the `depth` density function's gradient (now a separate `resonantcaves:overworld/depth` registry entry, -64/320 → -256/256, tracking the dimension's real bounds rather than a "virtual" mismatched range — the root cause of the earlier clipping), the ore-vein Y-band (-60/51 → -252/-141, bottom-relative), the deepslate transition (absolute 0/8 → -192/-184, bottom-relative), and several sea-level-anchored surface-rule decoration thresholds in swamp/mangrove swamp/windswept gravelly hills/wooded badlands biomes (60/62/63/74/97, each shifted by the same -63 as sea level; the unrelated legacy `absolute: 256` badlands terracotta-banding threshold was left untouched, confirmed independent of build height). The bedrock floor surface rule needed no change (`above_bottom`-anchored, already relative). Vanilla's cave-cheese-threshold tweak (`0.27` → `0.15`, denser caves) moved into Resonant World's noise settings only, since Vanilla World is meant to be genuinely vanilla. Cloud height (vanilla 192, 128 below the build cap) is handled in Java since it isn't data-driven per-dimension by default: a client-only `CloudHeightMixin` overrides `DimensionEffects.getCloudsHeight()` to return 128 — guarded by checking the *loaded world's* `DimensionType.minY()` rather than `DimensionEffects` instance identity, since both presets share the `"minecraft:overworld"` effects id. A common-side `WorldPresetsMixin` makes Resonant World the default pre-selected preset by mutating `WorldPresets.DEFAULT` (confirmed safe: it's assigned via a method call, not a compile-time constant, so it's read fresh at each call site rather than inlined). The new preset is made visible in the world-creation menu via an additive merge into vanilla's `data/minecraft/tags/worldgen/world_preset/normal.json` tag (tags merge across mods by default, no full-file override needed). `./gradlew build` is green after this change. **After an in-world playtest of Resonant World, its caves turned out to be flooded with lava almost everywhere.** The root cause was two hardcoded absolute-Y constants in vanilla Java code (not exposed via any noise_settings JSON, so the earlier data-only re-anchoring missed them): `NoiseChunkGenerator.createFluidLevelSampler`'s default fluid-type decision (`y < Math.min(-54, seaLevel) ? LAVA : WATER` — in vanilla only the bottom 10 blocks above its min_y of -64, but with Resonant World's floor at -256 this covered 202 of 256 blocks below sea level, the dominant effect) and `AquiferSampler.Impl.getFluidBlockState`'s lava-pocket-noise threshold (`fluidLevel <= -10`, vanilla's same 54-blocks-above-the-floor pattern, a smaller secondary effect). Both were fixed with new mixins — `NoiseChunkGeneratorLavaLevelMixin` (the real fix, `@ModifyConstant` re-deriving `-54` as `settings.generationShapeConfig().minimumY() + 10`) and `AquiferLavaThresholdMixin` (captures the dimension's `minimumY` via a constructor `@Inject`, then `@ModifyConstant`s `-10` to `minimumY + 54`) — both formulas reproduce vanilla's exact behavior at vanilla's bounds (-64 + 10 = -54, -64 + 54 = -10) while scaling correctly for any other dimension height. The old check (`hit.getType() == HitResult.Type.MISS`) therefore rejected every station, even ones directly underfoot, which in turn forced `BlockPos.findClosest`'s outward search to exhaustively scan the entire search-radius volume (worst case roughly 2 million block positions) on every single attack before giving up — the lag spike. Fixed by checking whether the raycast's resulting `BlockHitResult` is the station block itself (clear: nothing else was in the way) rather than requiring an outright miss. Separately, per the player's request, the station→wielder beam now also damages whatever it actually hits along that segment (via `ProjectileUtil.getEntityCollision` against the fixed station-to-wielder points, since that beam isn't aimed by the wielder's look direction the way the main attack ray is) — mirroring the already-confirmed "the beam damages whatever it hits first, not necessarily the intended target" rule, now applied to both beams instead of just the outgoing one. Finally, `EXTENDED_STATION_SEARCH_RADIUS` (the player's and golem's station search, independent of the wand's 64-block attack range) was tuned down from 64 to 32 blocks after the player observed continued lag at greater distances from any station — a direct mitigation of the same underlying cost (the worse case scan volume shrinks by 8x at half the radius), kept as a separate, easily-retunable constant from `MAX_RANGE`. **An attack sound effect was attempted and then reverted, deferred to a later session.** Three different `SoundEvent`s were tried via `Entity.playSound(SoundEvent, float, float)` (the same convenience method vanilla's own `AbstractSkeletonEntity.shootAt` uses for its bow sound) — `SoundEvents.ENTITY_GUARDIAN_ATTACK`, `SoundEvents.BLOCK_CONDUIT_ATTACK_TARGET`, `SoundEvents.BLOCK_RESPAWN_ANCHOR_CHARGE` — none audible in-world, despite a temporary diagnostic log line proving the call executed on every attack, and independent verification that `ServerWorld.playSound` properly broadcasts a real `PlaySoundS2CPacket` via `PlayerManager.sendToAround` (not a no-op) — the same code path vanilla's own `PlayerEntity.attack()` uses for its sword-sweep sound, which the player confirmed was audible in the same session (ruling out a muted category or environment/audio-routing issue). A `/playsound minecraft:block.respawn_anchor.charge player @s` test, run directly to isolate the mod's code from the asset/engine, *was* audible — pinning the discrepancy specifically to the difference between that command and the mod's `Entity.playSound` call, not to anything broader. The root cause is not yet found (an "ambient vs. one-shot sound" theory was proposed and explicitly rejected by the player); the sound code was fully reverted (no `SoundEvent` import, no `playSound` call in `WandAttacks.fire`) rather than left half-working, to revisit fresh in a later session. Separately, the radar-orb color (reverted to ender-pearl teal per the player's earlier request) was tuned further: the real `ender_pearl.png` texture was extracted from the vanilla client jar and sampled (average RGB `~20,88,77`, matching `DARK_RED/GREEN/BLUE` almost exactly), confirming the dark end of the breathing cycle was already accurate — but the bright end (`153,255,204`) sat at a visually distinct, more minty-green hue, so the orb only looked like ender pearl for half its cycle. `LIGHT_RED/GREEN/BLUE` was recomputed to the same ~170° hue as the dark end, just brighter and slightly less saturated (`106,235,213`), so the whole breathing range stays within the ender-pearl teal family. Verified compiling and a full `./gradlew build` after each fix; in-world `runClient` confirmation of whether the louder sound is finally audible, and whether the orb now reads as ender-pearl-colored throughout its cycle, is the natural follow-up. **Feature 10 (Village Core) is implemented and build-verified** — the energy network's third payoff: a craftable, tierless block (`resonantcaves:village_core`, `block/VillageCoreBlock`/`block/entity/VillageCoreBlockEntity`) that is a pure `EnergyStorage` sink (insert-only on every side, auto-discovered by the existing `CableNetworkTicker` exactly like a Station, no new discovery code) and founds an auto-expanding village wherever it's placed. Storage is finite (`CORE_CAPACITY = 60,000`) and deliberately not a long-term battery: once per Minecraft day (gated by `World.getTime()`, not `getTimeOfDay()`, so sleep-skipping doesn't distort the cadence), the Core pays upkeep for every tracked house (`baseDraw + perVillagerDraw × residents`, residents found by scanning loaded villagers in a box around the Core for a `MemoryModuleType.HOME` falling inside that house's footprint — no fixed UUID list, so vanilla-bred children entering spare beds and departed/converted/dead villagers both fall out automatically with zero bookkeeping), then either spends any leftover at or above `HOUSE_GROWTH_THRESHOLD = 50,000` on a new house/decoration (a random offset within a tight 16-block radius, validated by strict heightmap-flatness + an all-air footprint + no overlap with the Core or existing structures — real vanilla village NBT templates placed via `StructureTemplateManager`/`StructureTemplate.place(...)`, the same mechanism vanilla's own jigsaw villages use, just invoked directly without the jigsaw algorithm) or, on a deficit day, rolls pillager conversion for however many villagers' worth of upkeep went unpaid (`floor(unpaid / perVillagerDraw)` distinct villagers selected without replacement, each independently rolling 25% via `EntityType.PILLAGER.create(...)` placed with the same `isSpaceEmpty`-gated loop `CreeperPackSpawning` already uses, releasing the dying villager's 4 POI tickets — `HOME`/`JOB_SITE`/`POTENTIAL_JOB_SITE`/`MEETING_POINT` — before `discard()`, since `Entity.discard()` unlike `onDeath` does not free them automatically) — then unconditionally zeroes its storage, burning anything left over regardless of which branch ran. A new `VillageBiomeStyle` enum resolves the Core's actual biome (with commonsense substitutions for biomes outside vanilla's five dedicated village styles — badlands→desert, windswept/peak biomes→taiga — rather than defaulting everything to plains) to a full, jar-verified catalog of real template identifiers per style (houses, farms, animal pens, and profession buildings — included on purpose, since vanilla's own `FindJobSiteTask` claims unclaimed job-site POIs and assigns professions automatically with zero new code — plus decor pieces placed every 4th growth event instead of a house). Exactly 1 starter villager spawns per new house (vanilla's own bed-claiming AI assigns its `HOME` automatically); deliberately left un-suppressed vanilla breeding on top means multi-bed houses (several of the larger templates have 3+ beds) are an inherent, player-independent source of slowly rising demand given a working farm — the intended gentle, self-paced pressure, not a player-triggered one. A non-craftable `creative_village_core` debug variant (mirroring the Creative Station's distribution convention) self-fills to `CORE_CAPACITY` immediately before each daily checkpoint via a `beforeDailyCheckpoint()` hook, so the whole growth loop can be observed with no cable network at all. A new read-only `VillageCoreScreen` (no `ScreenHandler`, following the Monitor/Station GUI convention exactly — a payload says which `Screen` to open, periodic once-per-second pushes refresh it while open) shows house count, villager count, stored energy, and sustained/deficit status. This design went through several rounds of scope negotiation before implementation: an earlier draft additionally specified an autonomous villager AI goal that breaks and fills terrain to level building sites, real road generation between houses via vanilla's own street NBT templates, and power-funded house auto-repair — all explicitly deferred (not built) after the player flagged the combined design as too large a single feature to de-risk responsibly; the shipped scope is deliberately just the core economic loop (a strict flatness/clearance site search with no fallback when no site is found — growth simply skips that day) plus the two purely-cosmetic, low-testing-risk pieces (biome variety, decorations) the player specifically asked to keep. Verified compiling, a full `./gradlew build`, and a headless server smoke test (mod initializes, all new block/block-entity/recipe/payload registrations load, and a freshly generated world's spawn chunks tick with no exceptions); in-world `runClient` confirmation of the site-search success rate on real terrain, the growth/conversion pacing at the suggested starting constants, and the biome-style resolution is the natural follow-up.

## What this project is

**Resonant Caves** is a single Fabric mod for Minecraft Java Edition implementing a "Surface Survival" overhaul. The core fantasy: **mining is impossible**, so the game becomes about exploration, hunting, and infrastructure instead of digging.

- New worlds default to the **"Resonant World"** preset — a re-anchored coordinate system (sea level 0, build cap 256, world bottom -256) giving roughly double vanilla's below-sea-level cave volume as *traversal* space, not mineable rock, while keeping vanilla's proportions above sea level. A genuinely untouched **"Vanilla World"** preset is also selectable for comparison.
- Stone and deepslate are **unbreakable** (like bedrock); everything else — including stone variants (granite, diorite, andesite, tuff), dirt, gravel, and all ores — stays breakable, so resources remain accessible only where exposed in cave walls.
- A naturally-occurring unbreakable ore ("Resonant Ore") passively emits a small amount of RF — the mod's only energy *source*. Getting that energy where it's needed is now the player's own job: craftable cables and stations (Feature 8) move and store it, in place of relying on an external tech mod's infrastructure.
- That power has a visible payoff beyond the Energy Monitor's readout and the Resonant Wand's combat draw: a **Village Core** (Feature 10) grows an auto-expanding village — new houses, new villagers — funded by sustained daily power production, with villagers defecting to pillagers if the network can't keep up.

An earlier feature — large fleeing herds — was built, played, and then fully reverted at the player's request; see Feature 2 below for what's left of them in the repo.

## Hard scope discipline (read before adding anything)

This is a ~20-hour, AI-assisted, *first* mod. The design explicitly rejects features that become rabbit holes. **Do NOT build:**

- A custom chunk generator from scratch — modify vanilla `noise_settings` instead.
- Custom pathfinding or terrain-safety navigation — vanilla pathfinding is acceptable for v1.
- A rewritten mob AI system or true flocking/boids — fake herd behavior with large spawn groups + a strong flee radius.
- Migration/sleeping/predator-prey systems.

**Do:** keep implementation simple, lean on vanilla systems with small tweaks, and use existing APIs.

The `resonantcaves:creeper_attracting_blocks` tag (Feature 5) is a data file, not new infrastructure, so other tech mods can be supported by adding their block IDs to it, with no Java changes needed.

## The features

Features 1-4 were the original MVP; build those in order, with **worldgen last** since debugging it can consume the entire budget. Features 5-9 are post-MVP additions built after an in-world playtest. **Feature 2 was later fully reverted at the player's request** (see their entries below and "Current state" for what was undone and what survived).

1. **Unbreakable stone & deepslate** *(Easy — built ✓)* — `minecraft:stone` and `minecraft:deepslate` only; all other blocks (stone variants granite/diorite/andesite/tuff, dirt, gravel, ores) stay breakable. Implemented in two layers sharing `UnbreakableBlocks.isUnbreakable`: (a) `AbstractBlockStateMixin` forces `calcBlockBreakingDelta` to `0` so survival players get the **bedrock feel** — no cracking animation, no progress; (b) a `PlayerBlockBreakEvents.BEFORE` handler is an authoritative server-side veto. Creative players are exempt (they break via a separate instant path). Verified compiling + a headless server smoke test.
2. **Large fleeing herds** *(Easy — built, then reverted ✗)* — originally, `HerdSpawning` used Fabric's `BiomeModifications` API to rewrite spawn group sizes for sheep/cows/pigs/chickens/horses, and `FleeingAnimalsMixin` injected a `FleeEntityGoal` + `StayNearHerdGoal` into sheep/cow/pig/chicken via mixin. Fully reverted at the player's request: `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted, the `FleeingAnimalsMixin` mixins.json entry removed, and the `HerdSpawning.register()` call removed from `ResonantCaves.onInitialize()`. No replacement code — animal spawning and AI are pure vanilla again.
3. **Resonant Ore** *(Medium — built ✓)* — `ModBlocks` registers `resonant_ore` (stone variant) and `deepslate_resonant_ore`, both bedrock-hard via `AbstractBlock.Settings.strength(-1.0F, 3600000.0F)` — naturally unmineable in any game mode, no mixin needed (`UnbreakableBlocks.isUnbreakable` lists them too, for a single source of truth). Only `resonant_ore` gets a `BlockItem`/creative-tab entry, since neither block is ever obtainable in survival. `ResonantOreBlockEntity` rolls a single `generationRate` (0–15 RF/t) from a triangular distribution peaking at 4, seeded deterministically from the block's position (and persisted to NBT). `ResonantEnergyStorage` is a stateless, extract-only `team.reborn.energy.api.EnergyStorage` (refuses insertion; `getAmount`/`getCapacity`/`extract` all read `generationRate` directly) registered via `EnergyStorage.SIDED.registerForBlockEntity`. Worldgen is a `minecraft:ore` configured feature + placed feature (`data/resonantcaves/worldgen/...`, count 52, size 6, full-world-height `above_bottom: 0` to `below_top: 0` placement so generation rate is consistent at every Y level regardless of which world preset is active) registered with `BiomeModifications.addFeature` at `GenerationStep.Feature.UNDERGROUND_ORES` — roughly iron ore's rarity, may be buried (`discard_chance_on_air_exposure: 0.0`). Verified compiling + a headless server smoke test.
4. **Two world presets: "Resonant World" (default) and "Vanilla World"** *(Medium — built ✓)* — see "Current state" above for the full history and exact Y-value table. `data/minecraft/worldgen/noise_settings/overworld.json` / `density_function/overworld/depth.json` are no longer overridden, so **"Vanilla World"** (`minecraft:normal`) is genuinely stock vanilla generation. **"Resonant World"** (`resonantcaves:resonant_world`) is a new namespaced dimension type + noise settings re-anchored to `min_y: -256` / `sea_level: 0` / build cap `256`, registered as a `world_preset` and made the pre-selected default via `WorldPresetsMixin`. The two presets are display-renamed via lang-key overrides (`generator.minecraft.normal` → "Vanilla World", `generator.resonantcaves.resonant_world` → "Resonant World") and both appear in the main (non-shift) preset list via an additive tag merge. `./gradlew runClient` + in-world exploration of both presets (no clipping, sensible cloud height, normal-feeling caves/ores/deepslate/fluids in Resonant World; identical-to-vanilla in Vanilla World) is the natural follow-up.
5. **Creeper tech targeting** *(Easy — built ✓)* — Resonant Caves defines a block tag, `resonantcaves:creeper_attracting_blocks` (`data/resonantcaves/tags/block/creeper_attracting_blocks.json`), pre-populated with ~40 energy-infrastructure blocks (storage units, transformers, cables, solar panels, generators) as `required: false` entries — no Java changes are needed to add more entries to this tag. `CreeperTargetingMixin` injects a `SeekAndDetonateGoal` into every creeper's `initGoals` (priority 4, alongside `MeleeAttackGoal`): the goal scans a 24×6×24-block box around the creeper for the nearest tagged block (`BlockPos.findClosest`), paths to it with vanilla navigation, and once within ~1.5 blocks calls the creeper's public `ignite()` — the existing fuse/explosion code then runs unmodified, so detonations are ordinary power-3 creeper explosions. A 30-second give-up timer abandons unreachable targets (relevant since stone/deepslate are unbreakable, so a walled-off machine room may be unreachable). This makes energy infrastructure — which the player will use to draw RF from Resonant Ore — a standing threat that rewards defensive placement rather than a "guaranteed destroy" mechanic. Verified compiling + a headless server smoke test; in-world `runClient` testing is the natural follow-up.
6. **Creeper pack spawning** *(Easy — built ✓)* — `CreeperPackSpawning` listens for `ServerEntityEvents.ENTITY_LOAD`: any newly-spawned creeper becomes a pack "leader" and immediately spawns 9–14 more creepers within 3 blocks of it (each candidate position checked with `World.isSpaceEmpty` before placement), with all pack members — leader included — tagged `resonantcaves_creeper_pack` so a later chunk reload of an existing creeper doesn't spawn another pack around it. Combined with Feature 5, a pack converging on a player's energy-infrastructure setup means multiple independent `SeekAndDetonateGoal` instances can detonate near the same infrastructure at once, amplifying the "defend your base" threat. **History:** the original approach (`HerdSpawning`'s `growHerd` raising the creeper `SpawnSettings.SpawnEntry` group size to 10–15, plus a `CreeperTargetingMixin.getLimitPerChunk()` override to 15) was verified correct at the registry level but produced no visible packs in-world — vanilla's per-member ±5-block random-walk placement mostly fails in this mod's narrow cave tunnels — so it was reverted in favor of the clustered-spawn approach above. Verified compiling + a headless server smoke test; in-world `runClient` confirmation is the natural follow-up.
7. **Resonant Helmet** *(Medium — built ✓)* — `ModItems` registers a custom `resonantcaves:resonant` `ArmorMaterial` (gold-tier defense/enchantability/equip-sound/toughness/knockback-resistance, iron's durability multiplier → 165 durability) and the `resonantcaves:resonant_helmet` `ArmorItem`. While worn, `NightVisionRefresher` (server tick) keeps amplifier-0 Night Vision permanently topped up (capped to 0.1 screen-effect strength by the client-only `NightVisionStrengthMixin`, and hidden from the inventory effects panel by `StatusEffectIconMixin`), and `ResonantCavesClient` renders a "hostile mob radar": a manually-drawn, depth-test-disabled billboard quad (`textures/misc/radar_orb.png`) hovers at the center of every hostile mob within 128 blocks, visible through walls, at a constant on-screen size regardless of distance. Each orb slowly "breathes" — a 300-tick cycle drives both its opacity (15%–~78%) and a color fade between dark teal (ender pearl) and light bluish-green (vanilla `GLOW` particle), and a small wander offset synced to that same cycle. Both the breathing phase and wander shape are seeded per-mob from its UUID, so orbs from the same pack desync rather than pulsing in unison. Textures were originally hand-derived from the recipe's own ingredient colors via a Pillow remap of vanilla gold-armor textures, then later replaced by a remap of the vanilla **turtle shell helmet** textures (16x16 item icon + 64x32 armor layer): the bright-green scute-highlight pixels (hue ~121°) become a coral/salmon ramp matching Resonant Ore's palette, the duller olive-green shell-base/AO-shading pixels (hue ~133–143°) become a gold ramp, and the brown strap pixels are left untouched. **The helmet is no longer craftable** — `ZombieHelmetSpawning` listens for `ServerEntityEvents.ENTITY_LOAD` and gives plain `ZombieEntity` spawns (husks/drowned/zombie villagers/zombified piglins excluded via an exact `getClass() == ZombieEntity.class` check) a 5% chance to spawn already wearing the helmet, tagging every rolled zombie `resonantcaves_zombie_helmet_roll` so chunk reloads don't re-roll. The helmet's head-slot equipment drop chance is set to `2.0F`, mirroring vanilla's pillager-captain banner (`PatrolEntity`) — `MobEntity.dropEquipment` treats any drop chance `> 1.0F` as an unconditional drop, so the helmet always drops on death regardless of what killed the zombie (player, mob, fall, fire, etc.), not just player kills. Verified compiling + a headless server smoke test (recipe count back to baseline 3766, confirming the recipe removal and no other regressions); in-world `runClient` confirmation of spawn rate/feel is the natural follow-up.
8. **Energy infrastructure: cables, stations, a monitor, and a Creative Station** *(Medium-Hard — built ✓)* — 8 new blocks, implementing the design from `CHATGPT_ENERGY_BRAINSTORM.md` with several corrections (see "Current state"). `block/EnergyTier.java` is a shared `IRON`/`COPPER`/`GOLD` enum (worst-to-best) holding each tier's cable stats (minimum comfortable throughput, flat per-cable loss) and station leak rate. **Cables** (`iron_cable`/`copper_cable`/`gold_cable`) hold no energy themselves — `CableNetworkTicker` (adapted from Tech Reborn's MIT-licensed `CableTickManager`) runs once per tick per connected same-tier network: it gathers the network via BFS, finds every directly-adjacent non-cable `EnergyStorage` (sources that `supportsExtraction`, sinks that `supportsInsertion`), simulates total sink demand, pulls enough from sources to cover that demand after this tier's loss, and pushes the post-loss amount into the sinks, fairly split across multiple targets the same way Tech Reborn's `dispatchTransfer` does. `EnergyTier.lossFraction` compounds a flat per-cable loss across the network's size (a distance proxy, avoiding real pathfinding) with a steep penalty when a tick's demand falls under the tier's minimum throughput (`IRON`'s minimum is `0`, so it's never penalized beyond its flat rate — it's the cheap, no-frills tier). Visually, cables render as a thin connected wire (a multipart blockstate reusing vanilla's `north`/`east`/`south`/`west`/`up`/`down` `BooleanProperty`s, the same ones `FenceBlock`/`GlassPaneBlock` use), not a solid cube, with a matching thin `VoxelShape` hitbox (a small core plus one arm per connected direction, cached for all 64 connection combinations) — see "Current state" for the follow-ups that replaced the original `cube_all` model and fixed the cable's opacity/lighting and item icon. **Stations** (`iron_station`/`copper_station`/`gold_station`) are a single block with one output face: `Properties.HORIZONTAL_FACING`, set toward the player on placement like a furnace. `StationBlockEntity.getEnergyStorage(Direction side)` returns an extract-only `LimitingEnergyStorage` view (throttled by `targetOutput`) when `side == FACING`, and an insert-only view for every other side — both wrapping one shared `SimpleEnergyStorage`. Right-clicking the block opens a GUI (`StationScreen`) showing the live stored-energy reading and a text field for typing a precise `targetOutput` value (a "Max" button fills in unlimited). Storage has no capacity cap and no max-output cap; the only loss is a continuous per-tier leak (`StationBlockEntity`'s 20-tick cadence reduces `storedEnergy` by a percentage every second, fastest on `IRON`, slowest on `GOLD`) — players lose energy for *holding* it, not for moving it. (An earlier 2-tall lower/upper version, reusing vanilla's `DOUBLE_BLOCK_HALF`, was replaced by this single-block design — see "Current state".) The **Energy Monitor** (`energy_monitor`) is a thin, item-frame-like plate placed by right-clicking one face of an existing cable — `FACING` points away from that face (vanilla wall-torch convention, horizontal-only), and `MonitorBlockEntity.getAttachedCable()` reads `world.getBlockEntity(pos.offset(FACING.getOpposite()))` directly, the exact cable it's mounted on. Every 20 ticks it updates an ambient `STATUS` enum property (still a plain `BlockState`, synced just like a furnace's `lit`) and appends a sample into its throughput-history buffer; right-clicking additionally opens a `MonitorScreen` GUI with a line graph of that same cable's throughput sampled once per second over the last 3 in-game days (a 3,600-entry circular buffer persisted in the block entity's NBT). It breaks itself if the cable it's mounted on is removed. (An earlier version was a full block that scanned all 6 neighbors for any adjacent cable — see "Current state" for why that was replaced.) The **Creative Station** (`creative_station`) is a single-block, non-craftable, creative-inventory-only debug source (same distribution convention as vanilla's Light Block/Barrier) whose `EnergyStorage` materializes exactly `targetOutput` RF/t from nothing every tick, with no real storage and no insertion; it reuses `StationScreen` in a `creative` display mode that hides the meaningless stored-energy line in favour of an "unlimited" label. Cable/station textures are pulled directly from Tech Reborn's `1.21.1` branch (`copper_cable`/`gold_cable`/`tin_cable`→our iron tier, and the low/medium/high-voltage storage-unit `front`/`side` textures→our iron/copper/gold stations) under its MIT license (credited in `README.md`); the monitor's 4 status textures (idle/underutilized/optimal/overloaded) and the Creative Station's pink-recolored texture are self-authored/recolored via Pillow. `creeper_attracting_blocks.json` was rewritten from ~40 now-removed Tech Reborn IDs to the 7 real, defensible energy blocks (cables/stations/monitor) — the Creative Station, a debug tool, is deliberately left out of it. Verified compiling + a headless server smoke test (mod initializes, all new registries/recipes/the rewritten tag load, a generated world's spawn chunks tick with no exceptions); in-world `runClient` confirmation of the energy flow/loss feel and the monitor's at-a-glance readability is the natural follow-up.
9. **Resonant Wand** *(Medium-Hard — built ✓)* — see "Current state" above for the full design and the vanilla-mechanics verification behind it. A hitscan weapon (`resonantcaves:resonant_wand`, `item/ResonantWandItem`) fueled by `WandPowerSource`-implementing stations, with shared attack logic in `item/WandAttacks`. Usable by the player (right-click), spawned on some skeletons instead of a bow (`SkeletonWandSpawning`, `goal/WandAttackGoal`, `mixin/SkeletonWandMixin` — with a weak no-station fallback so skeletons stay a threat far from infrastructure), and equippable on a Snow Golem by right-clicking it (`goal/GolemWandSeekGoal`, `mixin/SnowGolemWandMixin` — no fallback, requires a visible nearby station). The golem's held wand renders via a new `client/render/feature/ResonantWandFeatureRenderer` + `mixin/SnowGolemRendererMixin`. Attacks spawn an `ELECTRIC_SPARK` particle line (wielder→target always, station→wielder when powered). Verified compiling, a full `./gradlew build`, and a headless server smoke test; in-world `runClient` confirmation of the feel/balance is the natural follow-up.
10. **Village Core** *(Medium-Hard — built ✓)* — see "Current state" above for the full design. A craftable, tierless block (`resonantcaves:village_core`, `block/VillageCoreBlock`/`block/entity/VillageCoreBlockEntity`) wired into the cable network like a Station; it is a pure insert-only `EnergyStorage` sink with finite, non-hoarding storage (`CORE_CAPACITY`, burned at the end of every Minecraft day). Once per day it pays upkeep for its houses (a dynamic census via villagers' `MemoryModuleType.HOME`, no fixed list), spends any single-day surplus at `HOUSE_GROWTH_THRESHOLD` on a new house/decoration from a biome-resolved (`block/VillageBiomeStyle`) catalog of real vanilla village NBT templates, and on a deficit day rolls pillager conversion for however many villagers' worth of upkeep went unpaid. A non-craftable `creative_village_core` debug variant self-fills before each daily checkpoint. Right-clicking opens a read-only `VillageCoreScreen` (house/villager/power vital stats), following the existing Monitor/Station no-`ScreenHandler` GUI convention. An autonomous villager terrain-leveling AI goal, road generation, and house auto-repair were explicitly scoped out of this pass after a deliberate de-risking discussion — see "Current state" for what was deferred and why. Verified compiling, a full `./gradlew build`, and a headless server smoke test; in-world `runClient` confirmation of site-search success rate, growth/conversion pacing, and biome-style resolution is the natural follow-up.

## Architecture & tech decisions

- **Loader:** Fabric (Fabric Loom + Fabric API). Start from the [Fabric Example Mod](https://github.com/FabricMC/fabric-example-mod) template.
- **Language/tooling:** Java, IntelliJ IDEA.
- **Minecraft version (pinned): 1.21.1**, with **Yarn mappings** (`yarn 1.21.1+build.3`), **Java 21**, Fabric Loader `0.19.3`, Fabric API `0.116.12+1.21.1`, Loom `1.16-SNAPSHOT`, Gradle `9.4.1`. We deliberately chose the established 1.21.1 over the newest Minecraft (26.x): version mismatch — not coding — is the biggest risk, and 1.21.1 has the deepest tutorial/example base, the most mature library support (including TeamReborn Energy), and well-settled tooling. We use **Yarn** rather than Mojang mappings (the current example-mod template defaults to Mojang) because the 1.21.1 tutorial/example ecosystem is overwhelmingly Yarn-based. **Match all docs/examples to 1.21.1 + Yarn.**
- **Loom plugin gotcha:** Loom 1.16 splits the Gradle plugin. The plain `net.fabricmc.fabric-loom` plugin does **not** expose the `mappings`/`modImplementation` DSL (it has a mappings-less workflow). For the classic Yarn setup (`mappings "net.fabricmc:yarn:…:v2"` + `modImplementation`), use **`net.fabricmc.fabric-loom-remap`** — which is what `build.gradle` declares.
- **Worldgen lives as datapack-style JSON bundled inside the mod jar** (noise settings, world preset, configured/placed ore features) — not a separate datapack and not a Java chunk generator. This keeps the deliverable a single mod file ("install mod and play") while minimizing Java code.
- **AI goal injection** uses SpongePowered Mixin.
- **Energy** uses the **TeamReborn Energy API** (the mature Fabric-native equivalent of Forge RF). The ore only *reports available extraction and refuses insertion* — energy availability is computed on demand, so no save data or sync is needed.

## Building & running

The project is scaffolded (Loom + Fabric API) and builds successfully. Standard workflow:

```bash
./gradlew genSources    # generate decompiled, Yarn-mapped Minecraft sources for reference
./gradlew runClient     # launch a dev client to test in-game
./gradlew runServer     # launch a dev server
./gradlew build         # compile + produce build/libs/resonantcaves-<ver>.jar
```

**Gradle must run on a JDK 21** (Loom compiles and runs Minecraft 1.21.1 against Java 21; there is no Gradle toolchain configured, so Gradle uses the JVM that launches it). Fedora 44 ships no JDK 21 package (only 25/26), so this repo uses a user-local **Temurin JDK 21** at `/home/namith/jdks/jdk-21.0.11+10`. Invoke Gradle with that JDK:

```bash
JAVA_HOME=/home/namith/jdks/jdk-21.0.11+10 ./gradlew build
```

(Or set IntelliJ's Gradle JVM to that JDK.) There is no test-suite convention yet; add one when tests are introduced.

## Development workflow

Build **one feature at a time**, fully closing the loop each time: verify it compiles → launch the game → test the behavior in-world → commit → move to the next feature. Get a working mod (unbreakable stone) *before* touching worldgen.
