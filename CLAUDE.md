# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

**Resonant Caves** is a single Fabric mod for Minecraft Java Edition implementing a "Surface Survival" overhaul. The core fantasy: **mining is impossible**, so the game becomes about exploration, hunting, and infrastructure instead of digging.

- New worlds default to the **"Resonant World"** preset — a re-anchored coordinate system (sea level 0, build cap 256, world bottom -256) giving roughly double vanilla's below-sea-level cave volume as *traversal* space, not mineable rock, while keeping vanilla's proportions above sea level. A genuinely untouched **"Vanilla World"** preset is also selectable for comparison.
- Stone and deepslate are **unbreakable** (like bedrock); everything else — including stone variants (granite, diorite, andesite, tuff), dirt, gravel, and all ores — stays breakable, so resources remain accessible only where exposed in cave walls.
- A naturally-occurring unbreakable ore ("Resonant Ore") passively emits a small amount of RF — the mod's only energy *source*. Getting that energy where it's needed is now the player's own job: craftable cables and stations (Feature 8) move and store it, in place of relying on an external tech mod's infrastructure.

An earlier feature — large fleeing herds — was built, played, and then fully reverted at the player's request; see Feature 2 below for what's left of them in the repo.

## Hard scope discipline (read before adding anything)

This is a ~20-hour, AI-assisted, *first* mod. The design explicitly rejects features that become rabbit holes. **Do NOT build:**

- A custom chunk generator from scratch — modify vanilla `noise_settings` instead.
- Custom pathfinding or terrain-safety navigation — vanilla pathfinding is acceptable for v1.
- A rewritten mob AI system or true flocking/boids — fake herd behavior with large spawn groups + a strong flee radius.
- Migration/sleeping/predator-prey systems.

**Do:** keep implementation simple, lean on vanilla systems with small tweaks, and use existing APIs.

The `resonantcaves:creeper_attracting_blocks` tag (Feature 5) is a data file, not new infrastructure, so other tech mods can be supported by adding their block IDs to it, with no Java changes needed.

## The features

Features 1-4 were the original MVP; build those in order, with **worldgen last** since debugging it can consume the entire budget. Features 5-9 are post-MVP additions built after an in-world playtest. **Feature 2 was later fully reverted at the player's request** (see their entries below and "Current state" for what was undone and what survived).

1. **Unbreakable stone & deepslate** *(Easy — built ✓)* — `minecraft:stone` and `minecraft:deepslate` only; all other blocks (stone variants granite/diorite/andesite/tuff, dirt, gravel, ores) stay breakable. Implemented in two layers sharing `UnbreakableBlocks.isUnbreakable`: (a) `AbstractBlockStateMixin` forces `calcBlockBreakingDelta` to `0` so survival players get the **bedrock feel** — no cracking animation, no progress; (b) a `PlayerBlockBreakEvents.BEFORE` handler is an authoritative server-side veto. Creative players are exempt (they break via a separate instant path). Verified compiling + a headless server smoke test.
2. **Large fleeing herds** *(Easy — built, then reverted ✗)* — originally, `HerdSpawning` used Fabric's `BiomeModifications` API to rewrite spawn group sizes for sheep/cows/pigs/chickens/horses, and `FleeingAnimalsMixin` injected a `FleeEntityGoal` + `StayNearHerdGoal` into sheep/cow/pig/chicken via mixin. Fully reverted at the player's request: `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted, the `FleeingAnimalsMixin` mixins.json entry removed, and the `HerdSpawning.register()` call removed from `ResonantCaves.onInitialize()`. No replacement code — animal spawning and AI are pure vanilla again.
3. **Resonant Ore** *(Medium — built ✓)* — `ModBlocks` registers `resonant_ore` (stone variant) and `deepslate_resonant_ore`, both bedrock-hard via `AbstractBlock.Settings.strength(-1.0F, 3600000.0F)` — naturally unmineable in any game mode, no mixin needed (`UnbreakableBlocks.isUnbreakable` lists them too, for a single source of truth). Only `resonant_ore` gets a `BlockItem`/creative-tab entry, since neither block is ever obtainable in survival. `ResonantOreBlockEntity` rolls a single `generationRate` (0–15 RF/t) from a triangular distribution peaking at 4, seeded deterministically from the block's position (and persisted to NBT). `ResonantEnergyStorage` is a stateless, extract-only `team.reborn.energy.api.EnergyStorage` (refuses insertion; `getAmount`/`getCapacity`/`extract` all read `generationRate` directly) registered via `EnergyStorage.SIDED.registerForBlockEntity`. Worldgen is a `minecraft:ore` configured feature + placed feature (`data/resonantcaves/worldgen/...`, count 52, size 6, full-world-height `above_bottom: 0` to `below_top: 0` placement so generation rate is consistent at every Y level regardless of which world preset is active) registered with `BiomeModifications.addFeature` at `GenerationStep.Feature.UNDERGROUND_ORES` — roughly iron ore's rarity, may be buried (`discard_chance_on_air_exposure: 0.0`). Verified compiling + a headless server smoke test.
4. **Two world presets: "Resonant World" (default) and "Vanilla World"** *(Medium — built ✓)* — `data/minecraft/worldgen/noise_settings/overworld.json` / `density_function/overworld/depth.json` are no longer overridden, so **"Vanilla World"** (`minecraft:normal`) is genuinely stock vanilla generation. **"Resonant World"** (`resonantcaves:resonant_world`) is a new namespaced dimension type + noise settings re-anchored to `min_y: -256` / `sea_level: 0` / build cap `256`, registered as a `world_preset` and made the pre-selected default via `WorldPresetsMixin`. The two presets are display-renamed via lang-key overrides (`generator.minecraft.normal` → "Vanilla World", `generator.resonantcaves.resonant_world` → "Resonant World") and both appear in the main (non-shift) preset list via an additive tag merge. All vanilla constants anchored to absolute Y coordinates (lava level, aquifer threshold, deepslate transition, cloud height, etc.) were recomputed by preserving their distance from the relevant reference point (sea level, world bottom) — two of these are Java-side-only and required `NoiseChunkGeneratorLavaLevelMixin` and `AquiferLavaThresholdMixin`; the cloud height adjustment is `CloudHeightMixin` (client-only, guards on `DimensionType.minY()`, not `DimensionEffects` identity).
5. **Creeper tech targeting** *(Easy — built ✓)* — Resonant Caves defines a block tag, `resonantcaves:creeper_attracting_blocks` (`data/resonantcaves/tags/block/creeper_attracting_blocks.json`), populated with the 7 mod energy-infrastructure blocks (cables/stations/monitor, `required: false` entries) — no Java changes are needed to add more. `CreeperTargetingMixin` injects a `SeekAndDetonateGoal` into every creeper's `initGoals` (priority 4, alongside `MeleeAttackGoal`): the goal scans a 24×6×24-block box for the nearest tagged block (`BlockPos.findClosest`), paths to it with vanilla navigation, and once within ~1.5 blocks calls the creeper's public `ignite()` — the existing fuse/explosion code runs unmodified. A 30-second give-up timer abandons unreachable targets (relevant since stone/deepslate are unbreakable).
6. **Creeper pack spawning** *(Easy — built ✓)* — `CreeperPackSpawning` listens for `ServerEntityEvents.ENTITY_LOAD`: any newly-spawned creeper becomes a pack "leader" and immediately spawns 4–9 more creepers within 3 blocks of it (each candidate position checked with `World.isSpaceEmpty` before placement), bringing total pack size to 5–10. All pack members — leader included — are tagged `resonantcaves_creeper_pack` so chunk reloads of an existing creeper don't spawn a second pack around it. The `ENTITY_LOAD` approach (not vanilla's group-size spawn mechanic) is necessary because vanilla's ±5-block random-walk placement for pack members mostly fails `canSpawn` in this mod's narrow, unbreakable-walled cave tunnels.
7. **Resonant Helmet** *(Medium — built ✓)* — `ModItems` registers a custom `resonantcaves:resonant` `ArmorMaterial` (gold-tier defense/enchantability/equip-sound/toughness/knockback-resistance, iron's durability multiplier → 165 durability) and the `resonantcaves:resonant_helmet` `ArmorItem`. While worn, `NightVisionRefresher` (server tick) keeps amplifier-0 Night Vision permanently topped up (capped to 0.1 screen-effect strength by the client-only `NightVisionStrengthMixin`, and hidden from the inventory effects panel by `StatusEffectIconMixin`), and `ResonantCavesClient` renders a "hostile mob radar": a manually-drawn, depth-test-disabled billboard quad (`textures/misc/radar_orb.png`) hovers at the center of every hostile mob within 128 blocks, visible through walls, at a constant on-screen size regardless of distance. Each orb slowly "breathes" — a 300-tick cycle drives both its opacity (15%–~78%) and a color fade between dark teal and light bluish-green, and a small wander offset synced to that same cycle. Both the breathing phase and wander shape are seeded per-mob from its UUID, so orbs from the same pack desync rather than pulsing in unison. Textures are a Pillow-recolored remap of the vanilla **turtle shell helmet** (16×16 item icon + 64×32 armor layer): bright-green scute highlights → coral/salmon; duller olive-green base/AO shading → gold; brown straps unchanged. **The helmet is not craftable** — `ZombieHelmetSpawning` gives plain `ZombieEntity` spawns (exact `getClass() == ZombieEntity.class` check, excluding husks/drowned/etc.) a 5% chance to spawn wearing the helmet, tagging every rolled zombie `resonantcaves_zombie_helmet_roll` so chunk reloads don't re-roll. The head-slot drop chance is `2.0F` — `MobEntity.dropEquipment` treats any drop chance `> 1.0F` as unconditional, so the helmet always drops regardless of kill cause.
8. **Energy infrastructure: cables, stations, a monitor, and a Creative Station** *(Medium-Hard — built ✓)* — 8 new blocks. `block/EnergyTier.java` is a shared `IRON`/`COPPER`/`GOLD` enum (worst-to-best) holding cable stats (minimum comfortable throughput, flat per-cable loss) and station leak rate. **Cables** (`iron_cable`/`copper_cable`/`gold_cable`) hold no energy themselves — `CableNetworkTicker` (adapted from Tech Reborn's MIT-licensed `CableTickManager`) runs once per tick per connected same-tier network: gathers the network via BFS, finds every directly-adjacent non-cable `EnergyStorage` (sources=`supportsExtraction`, sinks=`supportsInsertion`), simulates total sink demand, pulls enough from sources after applying `EnergyTier.lossFraction` (flat per-cable loss compounded across network size, with a steep extra penalty when throughput falls under the tier's minimum — `IRON`'s minimum is `0`, so it only pays the flat rate), and pushes the post-loss amount into the sinks via a fair split. Cables render as a thin wire: a multipart blockstate reusing vanilla's `north`/`east`/`south`/`west`/`up`/`down` `BooleanProperty`s (same ones `FenceBlock`/`GlassPaneBlock` use), with a matching thin `VoxelShape` hitbox (core + one arm per connection, cached for all 64 combinations). **Stations** (`iron_station`/`copper_station`/`gold_station`) are a single block with `Properties.HORIZONTAL_FACING` set toward the player on placement (like a furnace). `StationBlockEntity.getEnergyStorage(Direction side)` returns an extract-only `LimitingEnergyStorage` (throttled by `targetOutput`) when `side == FACING`, and an insert-only view for all other sides — both wrapping one shared `SimpleEnergyStorage`. Right-clicking opens `StationScreen`: live stored-energy, a line graph of stored energy over time (1h/6h/1d/3d timeframe buttons), a live rate-of-change line, and a text field to set `targetOutput`. Storage has no capacity cap; the only loss is a continuous per-tier leak every second. **Energy Monitor** (`energy_monitor`) is a thin item-frame-like plate placed by right-clicking any face of an existing cable. `FACING` points away from that face; `MonitorBlockEntity.getAttachedCable()` reads `world.getBlockEntity(pos.offset(FACING.getOpposite()))` — the one cable it's mounted on. Every 20 ticks it updates an ambient `STATUS`/`MonitorFlow` blockstate property (visible at a glance without opening a GUI: color=utilization level, arrow=energy flow direction) and appends a sample into a 3,600-entry circular buffer; right-clicking opens `MonitorScreen` with the same line-graph/timeframe-preset UI. Breaks itself if its cable is removed. Flow direction is the whole-network direction broadcast to every cable in the network by `CableNetworkTicker.handleCableTick` at the end of each tick — not a per-cable per-edge value, so all monitors on a straight run always agree. **Creative Station** (`creative_station`) materializes exactly `targetOutput` RF/t from nothing every tick; reuses `StationScreen` in a `creative` mode. Cable/station textures are from Tech Reborn's `1.21.1` branch (MIT, credited in `README.md`); monitor textures are self-authored. The Creative Station is deliberately excluded from `creeper_attracting_blocks.json`.
9. **Resonant Wand** *(Medium-Hard — built ✓)* — a hitscan weapon (`resonantcaves:resonant_wand`, `item/ResonantWandItem`) fueled by `WandPowerSource`-implementing stations, with shared attack logic in `item/WandAttacks`. Three wielders: **player** (right-click), **skeleton** (`SkeletonWandSpawning` + `goal/WandAttackGoal` + `mixin/SkeletonWandMixin` — has a low-damage no-station fallback so skeletons stay dangerous far from infrastructure; vanilla's `AbstractSkeletonEntity.updateAttackType()` already routes a non-bow mainhand to melee, so the mixin only adds `WandAttackGoal` at priority 3), **Snow Golem** (right-click with wand to equip via `useOnEntity`; `goal/GolemWandSeekGoal` + `mixin/SnowGolemWandMixin` — no fallback, requires a station in line of sight). Hitscan via `ProjectileUtil.getCollision`; energy cost scales with hit distance (`20 + 5×blocks` RF, up to 340 RF at 64 blocks); damage is a triangular roll (2–12, peak ~6.67). The station search uses `World.raycast` with `RaycastContext` to the station's center to verify a clear line of sight — checking `hit.getType() == BLOCK && hit.getBlockPos().equals(stationPos)` (not `== MISS`), because a ray to the center of a solid block always stops at the block's surface and never produces a true miss. Attacks spawn `ELECTRIC_SPARK` particles (wielder→target always; station→wielder when powered). Golem's held wand renders via `client/render/feature/ResonantWandFeatureRenderer` + `mixin/SnowGolemRendererMixin` (this project's first constructor-injection mixin).
10. **Village Core** *(Hard — built ✓)* — `resonantcaves:village_core` is a craftable energy-powered village builder (recipe: Emerald Block × 4 + Iron Block × 4 + Nether Star). Accepts RF from the cable/station network (insert-only `SimpleEnergyStorage`). Once per Minecraft midnight (`world.getTime() / 24000 > lastProcessedDay`), runs one accounting pass: (1) reconciles citizen UUID registry — deaths detected by absent entity within 2×SCAN_RADIUS, births by newly appearing entities within SCAN_RADIUS; (2) if not yet started, places the vanilla `village/plains/town_centers` pool piece near the core using `StructurePoolElement.generate()`; otherwise charges upkeep = `houseCount × 800 + villagerCount × 400` RF — if surplus, pays emeralds (1/house) and grows by one more jigsaw piece (`GROWTH_COST = 60,000` RF), if deficit converts a cluster of random villagers to pillagers (25% chance each); (3) burns all remaining energy; tracks statistics (births, deaths, spawns, peak population, days active, pieces placed, houses placed, pillagers converted) and appends to a per-day `PopulationHistory` ring buffer. Village expansion mirrors vanilla `StructurePoolBasedGenerator` but one piece per midnight instead of all at once: the core maintains a `List<JigsawConnector>` frontier (positions, facings, pool IDs, and names of open outward jigsaw connectors from all placed pieces); each midnight `processMidnight` retries `grow()` up to 25 times until a piece is successfully placed (a single attempt can hit a dead-end connector and return nothing); one connector is selected per attempt (weighted by inverse Manhattan distance to core: closer = higher weight) and a compatible piece is found from its target pool by iterating all elements × 4 rotations, checking `J.target == C.myName` and `JigsawBlock.getFacing(J.state) == C.facing.getOpposite()`, then placing with `placementPos = C.worldPos.offset(C.facing).subtract(J.pos())`. **Dead-end filter**: connectors targeting `minecraft:empty` or null/zero-element pools are discarded at add-time (both in `layFoundation` and `grow`) via `isDeadEndPool()` — roughly 70% of vanilla village connectors point to `minecraft:empty` and would otherwise waste attempts. **Overlap check**: uses `boundingBox.expand(-1)` against `entity.placedBoxes` to allow the 1-block border sharing that vanilla road/house junctions require; house pieces are additionally checked against `entity.placedHouseBoxes` (a separate list) using the unexpanded box to prevent wall-sharing between adjacent houses. **Terrain carving**: before `element.generate()`, `clearTerrain()` replaces natural terrain blocks (grass, dirt, sand, gravel, stone, stone variants, deepslate) in the full bounding box with air — necessary because vanilla pool elements ship with `BlockIgnoreStructureProcessor.AIR` which strips air from templates, so `generate()` only places solid blocks and leaves existing terrain blocks in interior positions; `clearTerrain` is our substitute for vanilla's BEARD_BOX "carve above" terrain-adaptation pass. **Foundation Y**: `findFoundationPos()` uses `getTopY(MOTION_BLOCKING_NO_LEAVES, x, z) - 1` (the top solid block, not the first air block). **Known limitation**: all pieces are placed at the connector's inherited Y (derived from the foundation), so the village stays at one elevation on flat terrain; per-piece Y adaptation (vanilla's behavior on slopes) is not implemented. Emerald storage is Storage Drawers-style: right-click holding emeralds = insert whole stack; right-click empty hand = extract 64; sneak+right-click = open GUI. Also registers via Fabric Transfer API (`ItemStorage.SIDED`) for hopper/pipe compatibility using an `EmeraldStorage` inner class extending `SnapshotParticipant<Long>`. **Block is unbreakable once placed** in survival (added to `UnbreakableBlocks.isUnbreakable()`). GUI (sneak+right-click): stats rows + emerald count + a `PopulationGraphPanel` with 3d/7d/28d/Max timeframe buttons. Jigsaw API: `RegistryKeys.TEMPLATE_POOL`; `JigsawBlock.getFacing(state)` (NOT `JigsawBlock.ORIENTATION`); `JigsawBlockEntity.POOL_KEY`/`NAME_KEY`/`TARGET_KEY`; `StructureLiquidSettings.APPLY_WATERLOGGING`. **Creative Village Core** (`resonantcaves:creative_village_core`): same block entity type, no RF requirement, adds "Simulate Day" button → sends `SimulateDayPayload` C→S → server calls `processMidnight(world, true)` immediately. Textures use vanilla `minecraft:block/emerald_block` (survival) and `minecraft:block/pink_concrete` (creative) as placeholders.

## Architecture & tech decisions

- **Loader:** Fabric (Fabric Loom + Fabric API). Start from the [Fabric Example Mod](https://github.com/FabricMC/fabric-example-mod) template.
- **Language/tooling:** Java, IntelliJ IDEA.
- **Minecraft version (pinned): 1.21.1**, with **Yarn mappings** (`yarn 1.21.1+build.3`), **Java 21**, Fabric Loader `0.19.3`, Fabric API `0.116.12+1.21.1`, Loom `1.16-SNAPSHOT`, Gradle `9.4.1`. We deliberately chose the established 1.21.1 over the newest Minecraft (26.x): version mismatch — not coding — is the biggest risk, and 1.21.1 has the deepest tutorial/example base, the most mature library support (including TeamReborn Energy), and well-settled tooling. We use **Yarn** rather than Mojang mappings (the current example-mod template defaults to Mojang) because the 1.21.1 tutorial/example ecosystem is overwhelmingly Yarn-based. **Match all docs/examples to 1.21.1 + Yarn.**
- **Loom plugin gotcha:** Loom 1.16 splits the Gradle plugin. The plain `net.fabricmc.fabric-loom` plugin does **not** expose the `mappings`/`modImplementation` DSL (it has a mappings-less workflow). For the classic Yarn setup (`mappings "net.fabricmc:yarn:…:v2"` + `modImplementation`), use **`net.fabricmc.fabric-loom-remap`** — which is what `build.gradle` declares.
- **Worldgen lives as datapack-style JSON bundled inside the mod jar** (noise settings, world preset, configured/placed ore features) — not a separate datapack and not a Java chunk generator. This keeps the deliverable a single mod file ("install mod and play") while minimizing Java code.
- **AI goal injection** uses SpongePowered Mixin.
- **Energy** uses the **TeamReborn Energy API** (the mature Fabric-native equivalent of Forge RF). The ore only *reports available extraction and refuses insertion* — energy availability is computed on demand, so no save data or sync is needed.
- **Two entrypoints:** `ResonantCaves` (both sides) and `ResonantCavesClient` (client only, declared in `fabric.mod.json`). Client-only code (radar rendering, mixin screen-effect patches) must live in the client entrypoint or client-only mixins to avoid crashing dedicated servers.
- **GUI networking — no `ScreenHandler`:** The station, monitor, and creative-station GUIs have no item slots, so they skip Minecraft's slot-sync machinery entirely. The server sends an `Open*ScreenPayload` when the player right-clicks, and the client opens a plain `Screen` subclass directly. While the screen is open, `ModNetworking` pushes a small `*UpdatePayload` every 20 ticks. New screens should follow this pattern — register payloads in `ModNetworking.register()`, receive on the client in `ResonantCavesClient.onInitializeClient()`.
- **Registration pattern:** `ModBlocks`, `ModItems`, and `ModBlockEntities` each hold `public static final` registry fields (initialized at class-load time) and a `register()` method that Loom calls once during `onInitialize()`. `ModBlockEntities.register()` also calls `CableNetworkTicker.register()`. Add new blocks/items/block-entities here; do not scatter registry calls across feature classes.

## Building & running

The project is scaffolded (Loom + Fabric API) and builds successfully. Standard workflow:

```bash
./gradlew genSources    # generate decompiled, Yarn-mapped Minecraft sources for reference
./gradlew runClient     # launch a dev client to test in-game
./gradlew runServer     # launch a dev server
./gradlew build         # compile + produce build/libs/resonantcaves-<ver>.jar
```

**Gradle must run on a JDK 21** (Loom compiles and runs Minecraft 1.21.1 against Java 21; there is no Gradle toolchain configured, so Gradle uses the JVM that launches it). Fedora 44 ships no JDK 21 package (only 25/26), so this repo uses a user-local **Temurin JDK 21** at `/home/namith/jdks/jdk-21.0.11+10`. The `JAVA_HOME` prefix is required for **every** Gradle invocation, not just `build`:

```bash
JAVA_HOME=/home/namith/jdks/jdk-21.0.11+10 ./gradlew build
JAVA_HOME=/home/namith/jdks/jdk-21.0.11+10 ./gradlew runClient
JAVA_HOME=/home/namith/jdks/jdk-21.0.11+10 ./gradlew runServer
```

(Or set IntelliJ's Gradle JVM to that JDK.) There is no test-suite convention yet; add one when tests are introduced.

## Development workflow

Build **one feature at a time**, fully closing the loop each time: verify it compiles → launch the game → test the behavior in-world → commit → move to the next feature. Get a working mod (unbreakable stone) *before* touching worldgen.

## Maintaining these docs

Three documents, three distinct audiences. Update the right one when its content changes — never add content to the wrong file.

| File | Audience | Contains | Update when |
|---|---|---|---|
| `README.md` | Players / contributors | What the mod does (feature list); how to build; credits | A feature becomes player-visible, or build requirements change |
| `CLAUDE.md` | Future Claude instances | *Current* code architecture, package map, scope discipline, implementation reference (class names, patterns, invariants) | The code architecture changes — **not** to record how it got there |
| `HISTORY.md` | Post-mortem / context | Chronological build log — decisions made, iterations, reversals, playtests | After each feature or session; **append-only**, never edit past entries |

**Cross-cutting rule:** if a sentence says "we tried X, then switched to Y," it belongs in `HISTORY.md`. `CLAUDE.md` describes what Y *is*; `README.md` describes what Y *does for the player*. When a feature is complete, update all three in the same commit.
