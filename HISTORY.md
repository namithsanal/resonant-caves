The project is scaffolded as a Fabric mod targeting **Minecraft 1.21.1** (pinned versions under "Architecture & tech decisions").
**Feature 1 (unbreakable stone & deepslate) is implemented and verified** — bedrock-like (no mining animation) via a mixin plus a server-side veto.
**Feature 2 (large fleeing herds) was implemented, verified, and later fully reverted at the player's request** — it originally raised biome spawn group sizes via Fabric's `BiomeModifications` API and injected a `FleeEntityGoal` into sheep/cow/pig/chicken via mixin (horses got larger herds but stayed tame); `HerdSpawning`, `FleeingAnimalsMixin`, and `StayNearHerdGoal` were deleted and the `FleeingAnimalsMixin` entry removed from `resonantcaves.mixins.json`, with no replacement code, so all passive animals are back to vanilla spawn-group sizes and behavior.
**Feature 3 (Resonant Ore) is implemented and verified** — two bedrock-hard ore blocks with a block entity that rolls a 0–15 RF/t generation rate and exposes it via the TeamReborn Energy API, generated underground at roughly iron ore's rarity via `BiomeModifications.addFeature`.
**Feature 4 (elevated world) went through several iterations and is now implemented as a separate, selectable world preset rather than an override of vanilla's Overworld.** Earlier versions overrode vanilla's actual `noise_settings/overworld.json` + `density_function/overworld/depth.json` directly — shifting sea level to 280, which clipped hills against the still-fixed Y320 build cap since the height envelope itself was never extended. The current design instead adds two coexisting world presets, chosen at world-creation time: **"Vanilla World"** (`minecraft:normal`, relabeled via a lang-key override) is genuinely untouched vanilla generation — `data/minecraft/worldgen/noise_settings/overworld.json` and `data/minecraft/worldgen/density_function/overworld/depth.json` are no longer overridden at all. **"Resonant World"** (`resonantcaves:resonant_world`, registered in `data/resonantcaves/worldgen/world_preset/`) is a new namespaced dimension type + noise settings (`data/resonantcaves/dimension_type/resonant_overworld.json`, `data/resonantcaves/worldgen/noise_settings/resonant_overworld.json`) re-anchored to a symmetric `min_y: -256` / `sea_level: 0` / build cap `256` coordinate system — chosen so the headroom above sea level matches vanilla's proportions almost exactly (256 vs vanilla's 257), eliminating the earlier clipping by construction rather than patching around it, while giving roughly double vanilla's below-sea-level cave volume. Both presets use the registry key `minecraft:overworld` for their Overworld dimension (that's how world presets work — picked once per save, not concurrent dimensions), so every other mod feature (unbreakable stone, Resonant Ore, creepers, the helmet) works identically in both without any changes. Several vanilla values anchored to specific Y coordinates were recomputed for Resonant World by preserving their *distance* from the relevant vanilla reference point, then re-anchoring to the new bounds: the ceiling terrain-dampening gradient (240/256 → 177/193, sea-level-relative), the floor solid-layer gradient (-64/-40 → -256/-232, bottom-relative), the `depth` density function's gradient (now a separate `resonantcaves:overworld/depth` registry entry, -64/320 → -256/256, tracking the dimension's real bounds rather than a "virtual" mismatched range — the root cause of the earlier clipping), the ore-vein Y-band (-60/51 → -252/-141, bottom-relative), the deepslate transition (absolute 0/8 → -192/-184, bottom-relative), and several sea-level-anchored surface-rule decoration thresholds in swamp/mangrove swamp/windswept gravelly hills/wooded badlands biomes (60/62/63/74/97, each shifted by the same -63 as sea level; the unrelated legacy `absolute: 256` badlands terracotta-banding threshold was left untouched, confirmed independent of build height). The bedrock floor surface rule needed no change (`above_bottom`-anchored, already relative). Vanilla's cave-cheese-threshold tweak (`0.27` → `0.15`, denser caves) moved into Resonant World's noise settings only, since Vanilla World is meant to be genuinely vanilla. Cloud height (vanilla 192, 128 below the build cap) is handled in Java since it isn't data-driven per-dimension by default: a client-only `CloudHeightMixin` overrides `DimensionEffects.getCloudsHeight()` to return 128 — guarded by checking the *loaded world's* `DimensionType.minY()` rather than `DimensionEffects` instance identity, since both presets share the `"minecraft:overworld"` effects id. A common-side `WorldPresetsMixin` makes Resonant World the default pre-selected preset by mutating `WorldPresets.DEFAULT` (confirmed safe: it's assigned via a method call, not a compile-time constant, so it's read fresh at each call site rather than inlined). The new preset is made visible in the world-creation menu via an additive merge into vanilla's `data/minecraft/tags/worldgen/world_preset/normal.json` tag (tags merge across mods by default, no full-file override needed). `./gradlew build` is green after this change.
**After an in-world playtest of Resonant World, its caves turned out to be flooded with lava almost everywhere.** The root cause was two hardcoded absolute-Y constants in vanilla Java code (not exposed via any noise_settings JSON, so the earlier data-only re-anchoring missed them): `NoiseChunkGenerator.createFluidLevelSampler`'s default fluid-type decision (`y < Math.min(-54, seaLevel) ? LAVA : WATER` — in vanilla only the bottom 10 blocks above its min_y of -64, but with Resonant World's floor at -256 this covered 202 of 256 blocks below sea level, the dominant effect) and `AquiferSampler.Impl.getFluidBlockState`'s lava-pocket-noise threshold (`fluidLevel <= -10`, vanilla's same 54-blocks-above-the-floor pattern, a smaller secondary effect). Both were fixed with new mixins — `NoiseChunkGeneratorLavaLevelMixin` (the real fix, `@ModifyConstant` re-deriving `-54` as `settings.generationShapeConfig().minimumY() + 10`) and `AquiferLavaThresholdMixin` (captures the dimension's `minimumY` via a constructor `@Inject`, then `@ModifyConstant`s `-10` to `minimumY + 54`) — both formulas reproduce vanilla's exact behavior at vanilla's bounds (-64 + 10 = -54, -64 + 54 = -10) while scaling correctly for any other dimension height. An initial version of the `NoiseChunkGeneratorLavaLevelMixin` handler had its parameter order backwards (Mixin requires the modified constant's type first, then captured target-method args — not the reverse), which crashed the dedicated server outright at boot; caught via a headless smoke test before being shipped. Verified compiling + a headless smoke test that exercises real chunk/aquifer generation (not just mod/mixin loading) for Resonant World; this only affects newly-generated chunks, so previously-explored saves keep their already-placed lava.

**Feature 6 (creeper pack spawning) is implemented and verified** — after two attempted fixes to a group-size-based approach (raising `SpawnSettings.SpawnEntry`'s creeper group size to 10–15 via `HerdSpawning`'s `growHerd`, then also overriding `CreeperEntity.getLimitPerChunk()` to 15) still produced no visibly larger packs in-world, a registry-level diagnostic (logged at server start, no playtest needed) confirmed both were correctly applied — every overworld biome had `creeper*(10-15):100` and `getLimitPerChunk()` returned 15 — so the real issue is vanilla's group-spawn algorithm itself: each pack member is placed via its own independent ±5-block random walk from the initial roll position, which mostly fails `canSpawn`'s space checks in this mod's narrow, unbreakable-walled cave tunnels. Both the group-size rewrite and the `getLimitPerChunk` override were reverted, and replaced with `CreeperPackSpawning`: on `ServerEntityEvents.ENTITY_LOAD`, any newly-spawned creeper becomes a pack "leader" and spawns 9–14 more creepers within 3 blocks of it, each candidate position checked with `World.isSpaceEmpty` before placement, with all pack members (including the leader) tagged `resonantcaves_creeper_pack` so chunk reloads of existing creepers don't repeatedly spawn new packs around them. Verified compiling + a headless server smoke test; in-world `runClient` confirmation of pack size/feel is the natural follow-up. `./gradlew build` is green and a headless server smoke test loads the mod + mixins + worldgen cleanly. After an in-world playtest, several follow-up refinements were made to Feature 7's Night Vision and radar effects. `NightVisionRefresher` was rewritten to track each player's helmet-wearing state in a `WEARING` set: Night Vision is now applied/removed the instant a player equips/unequips the helmet (previously there was a delay until the next 100-tick refresh), with `EFFECT_DURATION_TICKS` raised 220→320 — vanilla's `GameRenderer.getNightVisionStrength` pulses the screen-darkness fade once the remaining duration drops below 200 ticks, causing a visible "flicker" near each refresh, and 320 keeps the minimum remaining duration safely above that threshold. A new client-only `NightVisionStrengthMixin` caps the value returned by `GameRenderer.getNightVisionStrength()` to 0.1 for helmet wearers, making the helmet's Night Vision noticeably weaker/dimmer than the vanilla potion (tuned interactively: 0.5→0.15→0.05→0.1 final). A new client-only `StatusEffectIconMixin` (registered under the `client` list in `resonantcaves.mixins.json`, since `AbstractInventoryScreen` doesn't exist on a dedicated server) filters `AbstractInventoryScreen.drawStatusEffects`'s effect list down to `shouldShowIcon()`-true effects, so the inventory-screen effect panel no longer shows Night Vision (the top-right HUD already respected `showIcon=false` via `InGameHud`). Separately, `ResonantCavesClient` was rewritten so the radar is visible through walls: instead of spawning `ParticleTypes.GLOW` particles on `ClientTickEvents.END_CLIENT_TICK` (depth-tested and occluded by walls), it now hooks `WorldRenderEvents.AFTER_TRANSLUCENT` and draws a manual billboard quad per hostile mob via `RenderLayer.getTextSeeThrough(...)`, textured with a new 32x32 radial-gradient PNG (`assets/resonantcaves/textures/misc/radar_orb.png`). Since `getTextSeeThrough`'s "ALWAYS_DEPTH_TEST" phase is actually a no-op for `GL_ALWAYS` (Mojang special-cases `depthFunction == 519` to skip touching GL state), the draw is wrapped in explicit `RenderSystem.disableDepthTest()`/`enableDepthTest()` calls so the orb truly renders through walls. Each orb's quad size scales with distance to the camera (`ANGULAR_SIZE = 0.004`) so it stays a small, constant size on screen regardless of range; a shared 300-tick (~15s) breathing cycle drives both opacity (fading between 15% and ~78%) and color (fading between dark teal, matching an ender pearl, and a light bluish-green matching vanilla's `GLOW` particle); and each mob gets a UUID-seeded per-axis frequency/phase offset so its breathing and a small (±0.25 block) wander around its center are out of sync with other mobs' but synced with each other. **An in-world `runClient` playtest then found two real bugs and prompted one tuning change.** Standing directly on top of a powered station still reported "No power source in range," with a noticeable lag spike on every attack — both traced to the same root cause in `hasLineOfSight`: a block raycast from the wielder's eyes to the station's *center* can never report a miss when nothing is in the way, because the ray stops at the first solid surface it touches, which (with nothing blocking it) is the station's own face — the center point itself is inside solid geometry and is never reached. The old check (`hit.getType() == HitResult.Type.MISS`) therefore rejected every station, even ones directly underfoot, which in turn forced `BlockPos.findClosest`'s outward search to exhaustively scan the entire search-radius volume (worst case roughly 2 million block positions) on every single attack before giving up — the lag spike. Fixed by checking whether the raycast's resulting `BlockHitResult` is the station block itself (clear: nothing else was in the way) rather than requiring an outright miss. **Feature 10 (Village Core) is implemented and build-verified** — a craftable, energy-powered block that builds a plains village from scratch using vanilla's jigsaw pool system, one piece per midnight. The block (`resonantcaves:village_core`, crafted from emerald blocks + iron blocks + nether star) accepts RF from the cable/station network and, each Minecraft midnight, runs an accounting pass: reconcile the citizen-villager registry (scan for births/deaths by UUID, skipping unloaded chunks), charge upkeep (`houseCount × 800 RF + villagerCount × 400 RF`), convert clustered villagers to pillagers if short on energy (25% chance per at-risk villager, nearest-cluster selection), earn one emerald per house, then spend `60,000 RF` to grow one jigsaw piece — then burn all remaining stored energy for the day. The village is seeded from `village/plains/town_centers` on the first midnight: `VillageExpander.layFoundation()` tries 8 horizontal offsets from the Core's position (cardinal + diagonal, ±9–12 blocks, surface from `MOTION_BLOCKING_NO_LEAVES` heightmap, must be within ±20Y of the Core), then calls `element.generate()` with `StructureLiquidSettings.APPLY_WATERLOGGING` — the same path vanilla's worldgen uses. Subsequent midnights run `VillageExpander.grow()`: it picks an open frontier connector (weighted by inverse Manhattan distance to the Core — connectors closer to the Core are more likely to be extended, keeping growth compact), then iterates pool elements × 4 rotations looking for a jigsaw block J where `J.nbt.getString(TARGET_KEY).equals(connector.myName())` AND `JigsawBlock.getFacing(J.state()) == connector.facing().getOpposite()`, computing placement as `C.worldPos().offset(C.facing()).subtract(matchJ.pos())`, checking for bounding-box overlaps and unloaded chunks before committing, and rolling back frontier/placedBoxes mutations if `element.generate()` fails. House pieces are detected by `poolId.contains("/houses/")` — `village/plains/houses` is the vanilla pool name — and each triggers a villager spawn at the house's center surface position. A key implementation detail: `getStructureBlockInfos(manager, foundationPos, rotation, random)` (called in `layFoundation`) returns **world-space** positions, while `getStructureBlockInfos(manager, BlockPos.ORIGIN, rotation, random)` (called in `grow`) returns **element-local** positions; the placement formula's correctness hinges on this distinction. The `JigsawConnector` record (`worldPos, facing, poolId, myName`) is serialized to NBT and lives in the block entity's frontier list across sessions; its `toNbt()` and `fromNbt()` methods are `public` (required because `VillageCoreBlockEntity`, in `block.entity`, calls them from outside `VillageExpander`'s package). Citizen tracking uses a `Set<UUID>` persisted to NBT; midnight reconciliation compares this set against a live villager scan within `SCAN_RADIUS = 64` blocks, only pruning absent UUIDs if the entity's chunk is loaded (to avoid false deaths from unloaded chunks). Emerald storage is a `long storedEmeralds` with a Storage Drawers-style block interaction (right-click holding emeralds = insert, right-click empty hand = extract 64, sneak-right-click = open stats GUI) and a Fabric Transfer API `Storage<ItemVariant>` (inner `EmeraldStorage` extending `SnapshotParticipant<Long>`) for hopper/pipe compatibility, registered via `ItemStorage.SIDED`. The stats GUI (`VillageCoreScreen`) shows villager count, house count, pieces placed, days active, peak population, lifetime pillagers converted, lifetime deaths, stored emeralds, and a population-over-time line graph (`PopulationGraphPanel`, 500-entry circular buffer, "3d"/"7d"/"28d"/"Max" timeframe buttons) — opened via sneak-right-click and updated by periodic `VillageCoreStateUpdatePayload` pushes while open. The GUI follows the no-`ScreenHandler` pattern already established for Station/Monitor screens: a `OpenVillageCoreScreenPayload` (11 fields — exceeds `PacketCodec.tuple()`'s 8-field limit, so uses `PacketCodec.of()` with explicit encode/decode lambdas) opens the screen S→C, and `VillageCoreStateUpdatePayload` (4 fields, standard `tuple()`) pushes live updates. The **Creative Village Core** (`resonantcaves:creative_village_core`, pink, not craftable, creative-inventory only) shares the same block entity type and midnight logic but skips the RF requirement entirely; its GUI adds a "Simulate Day" button that sends `SimulateDayPayload` C→S, triggering `processMidnight()` immediately without waiting for real midnight. The survival `VillageCoreBlock` is added to `UnbreakableBlocks.isUnbreakable()` (checked via `instanceof VillageCoreBlock && !(instanceof CreativeVillageCoreBlock)`) so it cannot be removed once placed in survival — a creative admin with the Creative Village Core can still break it. Three compilation errors were caught and fixed: (1) `MapColor.GRASS` does not exist in Yarn 1.21.1 (fixed to `MapColor.PALE_GREEN`); (2) `JigsawConnector.toNbt()`/`fromNbt()` were package-private and inaccessible from `block.entity` (fixed to `public`); (3) `OpenVillageCoreScreenPayload` had a broken mixed `tuple()`/`MANUAL_CODEC` draft (fixed to a clean `PacketCodec.of()` throughout). Verified compiling, a full `./gradlew build`, and a headless server smoke test reaching "Done" with no exceptions; in-world `runClient` confirmation of village growth, energy accounting, GUI, and creative "Simulate Day" is the natural follow-up.
**In-world playtesting of Feature 10 found a series of bugs, addressed across several build/test iterations.** The first playtest surfaced four problems: structures were generating 1 block too high (roads had holes from displaced grass blocks), the GUI stats text was covered by the graph, a placed piece overwrote the Village Core block itself, and the Deaths stat was stored but never rendered. Root causes and fixes: `findFoundationPos()` was using `getTopY()` (first air block Y, one above the surface) instead of `getTopY() - 1` (the actual top solid block) — fixed. GUI layout was rearranged: stats rows compacted, timeframe buttons moved below the graph, peak label moved inside the graph background. `boundingBox.contains(corePos)` check added to `grow()` to reject pieces that would overwrite the core. Deaths stat added to a fourth stats row, displayed in red. The second playtest showed only 29 pieces placed over 100 simulated days, with no houses ever generated. Root causes: ~70% of frontier connectors point to `minecraft:empty` (vanilla's terminator pool), wasting almost every attempt; the strict 3D bounding-box overlap rejected the 1-block border sharing that road/house junctions require; and with one attempt per midnight, hitting a dead-end connector produced zero pieces for that day. Fixes: `isDeadEndPool()` helper discards connectors targeting `minecraft:empty` or null/empty pools at add-time in both `layFoundation` and `grow`; overlap check changed to `boundingBox.expand(-1)` to allow 1-block border sharing; `processMidnight` now retries `grow()` up to 25 times per midnight until a piece is placed; a separate `placedHouseBoxes` list (also persisted to NBT) tracks house bounding boxes with the full (non-expanded) box for strict house-to-house overlap checks, preventing adjacent houses from sharing a wall while still allowing road/house junctions. The third round reported houses with dirt/sand filling their interiors. Root cause: vanilla pool elements ship with `BlockIgnoreStructureProcessor.AIR` which strips air-block placement from the template — `element.generate()` therefore only places solid blocks (walls, floors, roofs) and leaves existing terrain blocks in all interior positions that the template recorded as air. Vanilla's worldgen uses BEARD_BOX terrain adaptation to pre-carve the footprint; our substitute is `clearTerrain()`, called immediately before `element.generate()`, which replaces natural terrain blocks (grass, dirt, sand, gravel, stone, deepslate, and stone variants) inside the bounding box with air. A Y-snap approach (shift each piece's floor to terrain height at the piece's center, ±6 blocks) was also attempted to address "all structures at the same Y-level" — but it caused pieces at different heights to bypass the 3D bounding-box overlap system, causing houses to stack on top of each other, and a `fillBelow()` pass created visible artificial dirt platforms; both were reverted. The Y-level limitation (all pieces placed at the connector's inherited Y from the foundation) is a known consequence of not implementing per-piece terrain adaptation, and is accepted for now.
